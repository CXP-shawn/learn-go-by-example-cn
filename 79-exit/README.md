# 第 79 关：退出（师兄带你学 Go）

## 🎯 这一关你会学到

- 调用 `os.Exit(code)` 可以让 Go 进程立刻以指定整数状态码退出，不会做任何额外清理。
- 和在 `main` 结尾写 `return` 的最大区别是：`os.Exit` 完全跳过所有已注册的 `defer`，一个都不会执行。
- Go 的 `main` 函数签名是 `func main()` 没有返回值，不像 C 可以 `return 1`，想传非零退出码只能显式调 `os.Exit`。
- 退出码 `0` 表示成功，非零表示失败，Shell 脚本、CI 流水线全靠这个判断上一步有没有跑挂。
- 大多数业务代码应该用 `error` 逐层返回、在 `main` 里统一处理并调 `os.Exit`，而不是在程序深处随手 `Exit` 把后路全断掉。

## 🤔 先想一个问题

师弟，你有没有在 Shell 脚本里写过 `echo $?` 查上一条命令退出码，然后用 `if [ $? -ne 0 ]` 决定要不要报警？这套机制在 CI 流水线里几乎无处不在。那天你问我，Go 程序怎么给 Shell 返回一个非零退出码，光在 `main` 里写 `return` 是没用的，因为 Go 的 `main` 根本没有返回值。这就是 `os.Exit` 存在的意义，也是我们 Go by Example 最后一关要聊的压轴话题。

## 📖 看代码

```go
// 使用 `os.Exit` 可以立即以给定的状态退出程序。

package main

import (
	"fmt"
	"os"
)

func main() {

	// 当使用 `os.Exit` 时 `defer` 将_不会_ 被执行，
	// 所以这里的 `fmt.Println` 将永远不会被调用。
	defer fmt.Println("!")

	// 退出并且退出状态为 3。
	os.Exit(3)
}

// 注意，不像例如 C 语言，Go 不使用在 `main` 中返回一个整数来指明退出状态。
// 如果你想以非零状态退出，那么你就要使用 `os.Exit`。
```

## 🔍 师兄给你逐行拆

### Exit 是立刻走人

好，最终关来了，兄弟，听好。

`os.Exit` 接受一个整数状态码参数，调用的那一刻，进程直接寿终正寝，函数永远不会返回给调用方，就像它压根儿没打算回来一样。

你可以这样理解：正常的程序退出，就像正常下班——先收拾桌面，跟同事道个别，把手头的事交接清楚，然后才走出公司大门。而 `os.Exit` 呢，是从工位直接蹦起来，夺门而出，连外套都不穿，门都不带关的那种。

这里有个很容易踩的坑：你要是在调用 `os.Exit` 之前，用 `defer` 注册了一条打印感叹号的语句，那个感叹号是永远不会出现在终端上的。`defer` 的执行时机是函数正常返回或者 `panic` 展开的时候，而 `os.Exit` 两样都不是，它直接绕过了整套清理机制，`defer` 队列就这样被整个抛弃了。

所以，兄弟，如果你的程序里有需要收尾的资源，别指望 `defer` 在 `os.Exit` 之后帮你兜底，要在调用它之前，自己老老实实做好清理。

### defer 为什么全被跳过

师弟，这个问题我见很多人踩过，今天给你讲清楚。Go 的 `defer` 机制本质上依赖函数正常执行到 `return`，或者 `panic` 沿调用栈向上回溯时，运行时才会依次触发当前函数注册的所有延迟调用。换句话说，`defer` 的执行权完全握在 Go 运行时手里，只要运行时还活着、还在正常走流程，`defer` 就能被调用到。

但是 `os.Exit` 完全是另一回事。它底层直接向操作系统发起 `exit` 系统调用，进程立刻终止，Go 运行时根本没有机会去遍历当前 goroutine 的 defer 栈，更别提执行了。所有你精心注册的资源释放、日志 flush、数据落盘操作，统统在瞬间蒸发，没有任何执行机会。

这里有个对比值得你记住：`panic` 虽然也是异常退出，但它仍然会沿着调用栈向上传播，沿途触发每一层函数里注册的 `defer`，你还有机会用 `recover` 捕获它、做善后处理。`os.Exit` 则完全不走这套逻辑，是真正意义上的硬退出。

特别提醒你一点，`log` 包里的 `Fatal`、`Fatalf`、`Fatalln` 这几个函数，内部实现就是先打印日志，然后直接调用 `os.Exit(1)`，和你手动写 `os.Exit` 没有任何区别。很多师弟在 `main` 函数里注册了 defer 关闭数据库连接或者 flush 缓冲区，然后某个地方调了 `log.Fatal`，结果那些 defer 全部被跳过，数据丢了还找不到原因。遇到需要清理资源的场景，请老老实实用 `panic` 加 `recover`，或者在退出前手动执行清理逻辑，不要依赖 defer 配合 `os.Exit` 使用。

### Go 的 main 为什么不能 return int

师弟，你刚从 C 转过来，肯定对这个问题一脸懵逼，我当年也是。

先说说老朋友 C。C 语言的 `main` 函数签名是 `int main()`，你可以 `return 0` 表示程序正常退出，`return 1` 表示出错，shell 脚本里直接拿这个退出码判断流程，这是几十年传下来的老传统，简洁又直接，很多系统工具都依赖这个约定。

但你到了 Go，打开编辑器一看，`main` 的签名是 `func main()`，没有返回值，写个 `return 1` 直接编译报错，是不是觉得特别别扭？别急着骂，这是 Go 设计者故意这么干的。

他们的理念是：退出码本质上是一个系统级行为，是程序和操作系统之间的约定，把它藏在函数返回值里显得太隐晦，不够直白。既然是系统调用，就老老实实用 `os` 包的 `Exit` 函数来表达，代码意图一目了然，不用猜。

具体规则很简单：`main` 函数正常跑完、自然结束，Go 运行时会替你调用系统级的 `exit(0)`，退出码就是零，表示成功。如果你想传非零退出码，就必须显式地调用 `os.Exit`，传入你想要的数字，绝对不会有任何魔法隐式发生。

这种设计最大的好处是可审计性。你想查程序有哪些退出路径，直接全局搜 `os.Exit` 就完事了，所有异常退出点全部暴露在代码表面，debug 和 code review 的时候都省了大量脑力，比藏在各层 `return` 里舒服多了。

### 生产实战怎么用

`os.Exit` 真正适合登场的场景其实并不多，师弟要记住这一点。最典型的两种情况是：程序启动阶段发现配置文件缺失、端口参数非法这类初始化校验失败，以及遇到了无法恢复的致命错误必须立刻中止进程。除此之外，日常业务逻辑里请让 `error` 老老实实一层一层地冒泡，一路传回 `main`，不要图省事在程序深处直接调 `Exit`，那样会跳过所有的 `defer` 清理逻辑，缓冲区没有 flush、数据库连接池没有关闭、临时文件没有删除，出了问题排查起来会非常痛苦。

正确的做法是在 `main` 里统一收口：先打印结构化日志，再 flush 缓冲，关闭连接池，做完所有清理之后，才最终调用 `os.Exit(1)` 退出。Go 社区有一个广泛认可的惯用模式，师兄建议你也养成这个习惯：写一个 `run` 函数，把真正的业务逻辑全部放进去，让它返回 `error`；`main` 函数只负责调用 `run`，拿到错误之后格式化打印，然后调 `os.Exit`。这样职责清晰，测试也方便，读代码的人一眼就能看懂程序的生命周期。

## 🏃 跑一下试试

直接用 `go run exit.go` 运行这段程序，你会发现终端里什么都没输出，程序瞬间结束。这正是 `os.Exit` 的核心行为：一旦调用，进程立刻终止，连 `defer fmt.Println("!")` 注册的那个感叹号也永远不会出现在屏幕上。很多初学者第一次见到这个现象都会愣一下，以为是代码写错了，其实这是完全符合预期的设计。

```shell
go run exit.go
```

运行完啥也没有。紧接着用下面这条命令查看刚才进程的退出码：

```shell
echo $?
```

```text
1
```

咦，明明代码里写的是 `os.Exit(3)`，为什么显示 `1`？这是因为 `go run` 自己也是一个包装进程，它在内部编译并启动子进程，子进程退出码为 3，但 `go run` 检测到子进程非零退出后，自己返回了 `1`。想看到真实的 `3`，需要先编译再执行二进制文件：

```shell
go build -o exit exit.go
./exit
echo $?
```

```text
3
```

这次 `echo $?` 终于如实输出了 `3`，与源码中的 `os.Exit(3)` 完全对应。在脚本或 CI 流水线中依赖退出码做判断时，务必用编译后的二进制，而不是 `go run`，否则会得到错误的码值，导致流程判断失准。

## 💡 师兄的碎碎念

**师兄碎碎念**

先聊聊退出码的惯用约定，这些约定来自 POSIX 和各大 shell 的长期实践，写工具时遵守它们能让使用者少踩坑。`0` 表示成功，这是唯一的成功码；`1` 是最通用的错误码，当你不知道该用哪个非零值时就用它；`2` 通常代表命令行参数或语法错误，很多标准工具（比如 `grep`、`diff`）都这样用；`126` 表示命令存在但无法执行（权限不够）；`127` 表示命令根本找不到；`128+N` 是被信号 N 杀死的惯例，比如被 `SIGKILL`（信号 9）杀掉就是 `137`，被 `SIGTERM`（信号 15）杀掉就是 `143`。在写 CLI 工具时给不同错误场景分配有意义的退出码，会让调用方的脚本好写很多。

第二个坑：`log.Fatal` 和 `log.Fatalln`。很多人把它们当成「会打印日志并停下来」的安全出口，却没意识到它们内部就是 `l.Output(...)` 加上 `os.Exit(1)`。这意味着同样不会触发任何 `defer`，数据库连接不会关闭，缓冲区不会 flush，文件锁不会释放。在程序初始化阶段用 `log.Fatal` 快速失败问题不大，但在业务逻辑深处调用就很危险。

生产代码的建议做法是：让 `error` 一层层冒泡回 `main`，在 `main` 里集中做所有清理工作——关连接、flush 缓冲、释放锁——最后再调 `os.Exit`。这样整个退出路径清晰可控，便于审查和测试。

最后区分 `panic` 和 `os.Exit`：`panic` 会沿调用栈向上传播，每一层的 `defer` 都会被执行，`recover` 也有机会介入；而 `os.Exit` 是操作系统级别的直接终止，Go 运行时根本来不及做任何收尾工作。二者都能终止程序，但适用场景截然不同，不要混用。
