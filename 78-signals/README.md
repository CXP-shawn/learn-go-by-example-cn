# 第 78 关：信号（师兄带你学 Go）

## 🎯 这一关你会学到

- Unix 信号是内核或其他进程给你的程序发的异步通知，`SIGINT`、`SIGTERM` 是最常见的两个，必须知道。
- Go 的 `os/signal` 包把信号统一建模成通道里的值，用起来和普通通道没有区别。
- 用 `make` 创建接收信号的通道时，缓冲容量必须大于等于 1，否则信号会被悄悄丢掉。
- 调用 `signal.Notify` 把你关心的信号和通道绑定，内核发来的信号就会被转发到这个通道里。
- 标准套路是单独开一个 goroutine 阻塞等信号，主 goroutine 用 `done` 通道等待它通知自己退出。
- 这套写法就是 Go 服务优雅关闭的标准模板，容器部署、gRPC 服务全都这么用。

## 🤔 先想一个问题

你有没有遇到过按下 `Ctrl+C` 程序就直接崩掉、数据没刷盘、连接没关干净的惨剧？或者 `docker stop` 发出 `SIGTERM`，容器里的服务毫无反应，最后被 `SIGKILL` 强杀？在 Kubernetes 里 `kubectl delete pod` 也是同样的故事，给你 30 秒优雅退出的机会，但你的代码根本没听到信号。Go 的信号处理机制就是为了解决这些问题而生的，学完这一关你就能写出真正能优雅关闭的服务。

## 📖 看代码

```go
// 有时候，我们希望 Go 可以智能的处理 [Unix 信号](http://en.wikipedia.org/wiki/Unix_signal)。
// 例如，我们希望当服务器接收到一个 `SIGTERM` 信号时，能够优雅退出，
// 或者一个命令行工具在接收到一个 `SIGINT` 信号时停止处理输入信息。
// 我们这里讲的就是在 Go 中如何使用通道来处理信号。

package main

import (
	"fmt"
	"os"
	"os/signal"
	"syscall"
)

func main() {

	// Go 通过向一个通道发送 `os.Signal` 值来发送信号通知。
	// 我们将创建一个通道来接收这些通知。请注意，这个通道应该被缓存。
	sigs := make(chan os.Signal, 1)

	// `signal.Notify` 注册给定的通道，用于接收特定信号。
	signal.Notify(sigs, syscall.SIGINT, syscall.SIGTERM)

	// 我们可以在 main 函数中从 `sigs` 接收。
	// 但让我们看看如何在单独的 goroutine 中完成此操作，
	// 以演示更实际的正常关闭方案。
	done := make(chan bool, 1)

	go func() {
		// 这个协程执行一个阻塞的信号接收操作。
		// 当它接收到一个值时，它将打印这个值，然后通知程序可以退出。
		sig := <-sigs
		fmt.Println()
		fmt.Println(sig)
		done <- true
	}()

	// 	程序将在这里进行等待，直到它得到了期望的信号
	// （也就是上面的协程发送的 `done` 值），然后退出。
	fmt.Println("awaiting signal")
	<-done
	fmt.Println("exiting")
}
```

## 🔍 师兄给你逐行拆

### 信号到底是个啥

Unix 信号是操作系统内核提供的一种进程间异步通知机制，说白了就是内核或者另一个进程给你这个进程发一条「小纸条」，告诉你发生了某件事，你最好去处理一下。这个机制跟网络请求、文件读写完全不同——它随时可能打断你，不管你当前在干嘛，内核都会强行把信号塞进你的处理队列里。你可以把它想象成公司大群里老板突然 @ 你，不管你正在写代码还是上厕所，手机一震你就得放下手头的事情跑去看是什么情况，完全没有商量的余地。

常见的信号有那么几个，师弟你最好烂熟于心。用户在终端按下 `Ctrl+C`，内核就给前台进程发 `SIGINT`，表示「用户要中断你」；用 `kill` 命令不带参数默认发的是 `SIGTERM`，这是一种「礼貌性的请求终止」，进程可以捕获它、做完清理再退；但如果你用 `kill -9`，那发的就是 `SIGKILL`，这个信号进程既无法捕获也无法忽略，内核直接强杀，毫不留情，什么清理都来不及做。`SIGHUP` 历史上表示终端断开，现代程序里常被用来触发「重新加载配置文件」这个约定俗成的动作。`SIGSEGV` 则是段错误，程序访问了不该访问的内存地址，内核会发这个信号，通常意味着你代码写了 bug 或者有指针越界问题，见到它就老老实实去查内存访问逻辑吧。信号的种类其实还有很多，但上面这几个是日常开发中最高频碰到的，务必记牢。

### Go 用通道接信号

Go 语言在设计上把信号处理这件事跟通道（channel）深度绑定，思路非常优雅：信号本质上是「某个异步事件到来了」，而通道天生就是 Go 里处理异步事件的标准工具，两者一拍即合。具体怎么做呢？你先用 `make` 创建一个元素类型为 `os.Signal` 的通道，然后调 `signal.Notify`，把你关心的那些信号「登记」进去，之后运行时一旦收到这些信号，就会把对应的 `os.Signal` 值投递到你的通道里，你在代码里 receive 这个通道就能拿到信号了。整个流程干净利落，没有回调函数乱飞，代码读起来跟普通的通道操作一模一样。

这里有一个非常重要的细节，师弟你千万别踩坑——`make` 这个通道的时候缓冲容量**必须大于等于 1**，也就是说你要用 `make(chan os.Signal, 1)` 而不是 `make(chan os.Signal)`。原因是 Go 运行时往通道里投递信号这个动作是非阻塞的，如果通道已经满了或者是个无缓冲通道而接收方还没准备好，运行时不会等，直接把这次信号丢掉，就当没发生过。你可以把这个缓冲想象成宿舍楼下的收发室，快递来了先帮你存一个格子，等你有空了再下楼去取；要是格子满了或者收发室根本没格子，快递员就只好把快递带走，你啥也收不着。实际生产里建议缓冲容量根据你预期会同时到达的信号数量来设，但通常设成 `1` 对绝大多数场景已经够用了，因为信号处理的频率一般不会高到通道来不及消费的程度。另外别忘了在合适的时机调 `signal.Stop` 停止转发，否则通道和 goroutine 会一直挂在那里，算是轻微的资源泄漏。

### 两个 goroutine 的分工

光知道怎么创建信号通道还不够，代码结构怎么组织也很有讲究，来看看最经典的示例写法是什么套路。主要角色是两个通道：一个是刚才说的信号通道 `sigs`，另一个是用来传递「完成」通知的 `done` 通道，类型一般是 `chan bool`。

程序启动后，`main` 先打印一行 `awaiting signal`，表示自己已经准备好了、在等待信号到来；然后 `main` 并不直接去阻塞读信号通道，而是单独起一个 goroutine 专门负责这件事。这个子 goroutine 里面是一个阻塞的 receive 操作，一直等 `sigs` 通道有值进来；信号一到，它把信号的名字打印出来（通常是 `sig.String()` 的输出），然后往 `done` 通道里写一个 `true`，表示「我处理完了，可以退出了」；与此同时，`main` goroutine 一直阻塞在 `done` 通道的 receive 上，等到读到值，就打印 `exiting` 然后正常退出。

这种「主 goroutine 等完成信号、子 goroutine 处理业务信号」的分工方式有个很大的好处：子 goroutine 里面可以做任意的清理动作，比如刷缓冲区、关数据库连接、等正在处理的请求跑完，这些逻辑跟 `main` 的退出逻辑解耦得非常干净。你要是把信号处理逻辑全堆在 `main` 里，代码很快就会变成一坨面条，维护起来痛苦死了，所以这个拆分习惯要从一开始就养成。

### 生产实战建议

好了，理论和基础示例都讲完了，最后说说真正上生产环境的时候你要注意的三件事，这些是我踩过坑之后总结出来的经验，少走弯路。

第一件事，把信号处理跟 `context` 绑起来。现代 Go（1.16 及以后）直接提供了 `signal.NotifyContext` 这个函数，调一次就能返回一个 `ctx` 和一个 `cancel`，只要收到指定信号，这个 `ctx` 就自动变成 Done 状态，下游所有拿着这个 `ctx` 的组件都会感知到，联动取消干净利落，比手动维护通道优雅太多，强烈推荐直接用这个。

第二件事，优雅关闭务必设超时兜底。你在收到 `SIGTERM` 之后做清理，但清理本身也可能卡住或者超时，如果你不设一个兜底的最大等待时间，进程就会永远挂在那里，最终还是被外部 `SIGKILL` 暴力干掉，效果跟没有优雅关闭一样差。通常的做法是用 `context.WithTimeout` 给清理流程设一个时限，比如 30 秒，超时就强制退出，保证进程总能在有限时间内结束，这样既给了旧请求足够的时间跑完，又不会无限拖着。

第三件事，容器化场景下这件事更加关键，你要对它保持足够的重视。`docker stop` 命令和 Kubernetes 的 `preStop` 钩子，在触发停止逻辑时都是先发 `SIGTERM` 给容器主进程，等待一段宽限期（grace period，Kubernetes 默认 30 秒），如果进程还没退出才升级成 `SIGKILL` 强杀。这意味着你的 Go 服务只要正确响应了 `SIGTERM`，在宽限期内完成清理并退出，就能实现真正的零停机滚动部署，新旧实例交替时不会有请求被粗暴中断。反过来如果你的服务根本没捕获 `SIGTERM`，宽限期一到直接被 `SIGKILL`，正在处理到一半的请求就全报错了，用户体验极差，上线时运维同学也会来找你麻烦的。

## 🏃 跑一下试试

执行下面的命令启动程序，它会立刻打印 `awaiting signal` 并阻塞，耐心等待系统信号的到来。

```shell
go run signals.go
```

此时在终端按下 `Ctrl+C`，操作系统会向进程发送 `SIGINT`，程序捕获后打印信号名称，再打印 `exiting`，然后干净退出，典型输出如下。

```text
awaiting signal
interrupt
exiting
```

如果想验证 `SIGTERM` 是否同样有效，可以另开一个终端，先用 `ps aux | grep signals` 找到进程 PID，再执行 `kill <PID>` 发送默认的 `SIGTERM`，程序同样会打印信号名称并优雅退出，输出如下。

```text
awaiting signal
terminated
exiting
```

两种信号都能触发同一套退出流程，说明我们的多信号订阅写法完全生效，程序行为符合预期。

## 💡 师兄的碎碎念

**师兄碎碎念**

缓冲通道容量写 `1` 是硬性要求，绝对不能写 `0`。Go 运行时在向通道发送信号时不会阻塞等待接收方，如果通道是无缓冲的，运行时发现发不进去就直接丢弃，信号就这样悄无声息地消失了，`signal.Notify` 的签名故意没有默认值，就是让你亲手填这个数字，逼你意识到这个细节。

SIGKILL 和 SIGSTOP 在任何语言、任何框架里都捕获不到，这是操作系统的强制保留，内核级别的设计决定，Go 自然也不例外。如果你在 `signal.Notify` 的参数里加上这两个，程序不会报错，但永远也收不到它们，千万不要寄希望于用这两个信号做优雅退出。

优雅关闭有一套标准模板值得背下来：收到 SIGTERM 之后，第一步停止接收新请求或新任务，第二步等待已经在执行的旧任务跑完，第三步设置一个超时兜底，超时之后不管旧任务有没有完成都强制退出，防止进程挂死。

当你不再关心某个通道上的信号时，记得调用 `signal.Stop(ch)` 取消订阅，这样运行时就不会再往那个通道塞东西，也能让 GC 顺利回收相关资源。

现代 Go 更推荐 `signal.NotifyContext`，一行代码就能把信号订阅和 context 取消绑定在一起，信号一来 context 自动取消，所有下游 goroutine 都能感知到，比手写通道优雅很多，新项目优先选这个方案。
