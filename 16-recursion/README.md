# 第 16 关：递归（师兄带你学 Go）

## 🎯 这一关你会学到

这一关你将彻底搞懂「函数自己调用自己」这件听起来玄乎、其实超有规律的事。你会学会写经典的阶乘递归，还会学到 Go 里匿名函数想递归调自己必须先 `var` 声明的骚操作，以及斐波那契的朴素实现为什么会让 CPU 哭泣。

## 🤔 先想一个问题

你玩过那种游戏吗？打开一个宝箱，里面躺着另一个宝箱；打开那个，又是一个宝箱；再打开……直到最深处终于出现了一枚金币，游戏才宣布「任务完成」，然后你开始一层层退出，把沿途记录的东西带回起点。这就是递归最直白的样子：**一件事情可以拆成「做一小步 + 对更小规模的同样的事情重复」，直到遇到一个最简单到不需要再拆的情况为止——这种「函数在执行过程中直接或间接调用自身」的编程技巧，就叫做递归（Recursion）。**

## 📖 看代码

```go
// Go 支持 <a href="http://zh.wikipedia.org/wiki/%E9%80%92%E5%BD%92_(%E8%AE%A1%E7%AE%97%E6%9C%BA%E7%A7%91%E5%AD%A6)"><em>递归</em></a>。
// 这里是一个经典的示例。

package main

import "fmt"

// `fact` 函数在到达 `fact(0)` 前一直调用自身。
func fact(n int) int {
	if n == 0 {
		return 1
	}
	return n * fact(n-1)
}

func main() {
	fmt.Println(fact(7))

	// 闭包也可以是递归的，但这要求在定义闭包之前用类型化的 `var` 显式声明闭包。
	var fib func(n int) int

	fib = func(n int) int {
		if n < 2 {
			return n
		}
		return fib(n-1) + fib(n-2)

		// 由于 `fib` 之前在 `main` 中声明过，
		// 因此 Go 知道在这里用 `fib` 调用哪个函数。
	}

	fmt.Println(fib(7))
}
```

## 🔍 师兄给你逐行拆

### ① 递归三要素：基准条件、递归式、每次向基准收敛

师兄先给你立三根柱子，记住这三根柱子，你写递归就永远不会迷路。

**第一根柱子：基准条件（Base Case）。** 这是递归里最重要的一句话，也是新手最容易忘的一句话。基准条件就是「我不再往下拆了，直接给答案」的那个出口。拿宝箱比喻，就是「这个箱子里放的是金币，不是新箱子」的那一刻。在阶乘里，它就是 `n == 0` 时返回 `1`。没有基准条件会怎样？函数就会无限地调用自己，直到 Go 运行时把调用栈撑爆，然后你会看到一行让人头皮发麻的输出：`runtime: goroutine stack exceeds 1000000000-byte limit`，紧接着程序崩溃。这就是传说中的「栈溢出」——所以，写递归，第一件事永远是先想清楚出口在哪里。

**第二根柱子：递归式（Recursive Case）。** 就是「把大问题拆成小问题」的那一步。阶乘里是 `n * fact(n-1)`，含义是：我不知道 `n!` 是多少，但我知道它等于 `n` 乘以 `(n-1)!`，而 `(n-1)!` 嘛，就交给自己的分身去算。

**第三根柱子：每次调用必须向基准条件收敛。** 这是最容易被忽视但逻辑上最关键的一条。每次递归调用，传进去的参数必须离「出口」更近一步，否则你就陷入死递归了。`fact(n-1)` 每次把 `n` 减一，总有一天会到达 `0`，这就是收敛。如果你写了个 `fact(n+1)`，那恭喜你，你成功写出了一个优雅的死循环。生活类比：就像你剥洋葱，每剥一层，洋葱就小一圈，总有剥完的那天；你不能越剥越大，那就不是洋葱了，那是魔法道具。

### ② fact 阶乘展开：一层层下去，再一层层弹回来

我们来把 `fact(3)` 的执行过程像慢动作一样拆开看。

**展开阶段（问题下钻）：**

```
fact(3)
  → 3 * fact(2)
        → 2 * fact(1)
              → 1 * fact(0)
                    → 直接返回 1   ← 基准条件触发！
```

这个阶段，每次调用都把「我还没算完，等子问题给我答案」这件事压到调用栈上。调用栈就像一叠餐盘，每压一个进去就叠高一层。`fact(3)` 压进去，`fact(2)` 压进去，`fact(1)` 压进去，`fact(0)` 进去直接返回 `1`，不再往下压了。

**回弹阶段（答案上浮）：**

```
fact(0) 返回 1
fact(1) = 1 * 1 = 1，返回 1
fact(2) = 2 * 1 = 2，返回 2
fact(3) = 3 * 2 = 6，返回 6
```

这个阶段，餐盘一个个从顶部取走，每取一个就完成一次乘法，最终 `fact(3)` 拿到了答案 `6`。

整个过程就像你对着两面镜子中间站着，镜子里有无数个自己，但因为你每次离最近那面镜子远了一点点，最终在某个距离上影像消失了（基准条件），然后光线开始往回走，最终抵达你的眼睛（答案回弹）。

**容易踩的坑：** 递归深度 = 调用栈帧数。`fact(7)` 才 8 层，完全没压力。但如果你拿递归去算 `fact(100000)`，Go 的栈虽然会自动扩容，但最终还是有上限，而且大量的栈帧开销会让性能很难看。这时候你应该考虑迭代（用循环）或者尾递归优化——不过 Go 编译器目前不做尾调用优化，这是个小遗憾，你知道就好。

### ③ 匿名闭包递归：为什么必须先 var 再赋值，:= 为什么不行

这是 Go 里一个让初学者摔跟头的细节，师兄给你掰清楚。

**先看「正确写法」：**

```go
var fib func(n int) int
fib = func(n int) int {
    if n < 2 { return n }
    return fib(n-1) + fib(n-2)
}
```

**再看「直觉写法」（会编译报错）：**

```go
fib := func(n int) int {
    if n < 2 { return n }
    return fib(n-1) + fib(n-2)  // ← 这里 fib 未定义！
}
```

为什么 `:=` 不行？因为 `:=` 是「声明并赋值」的语法糖，它的执行顺序是：**先对右侧表达式求值，再把结果绑定到左侧变量名上**。右侧表达式是一个函数字面量，函数体里用到了 `fib` 这个名字，但此时左侧的 `fib` 还没有被放进当前作用域——它甚至还没出生呢！于是 Go 编译器直接报错：`undefined: fib`。

生活类比：就像你想雇一个人去帮你介绍他自己，但你还没给他发工牌、告诉他他叫什么，他当然不知道自己是谁。你得先办好入职手续（`var fib func(n int) int`，这一步把名字和类型注册进作用域），再让他上岗干活（`fib = func(...){...}`，这一步把具体逻辑交给他）。

`var fib func(n int) int` 这一行做了什么？它在当前作用域里声明了一个名为 `fib`、类型为 `func(n int) int` 的变量，初始值是 `nil`。关键是，**这个名字从此刻起就在作用域里了**。接下来赋值的函数字面量在编译时可以合法地引用 `fib` 这个名字，因为到函数真正被调用的时候，`fib` 已经指向了那个函数本身，一切就闭合了。

**容易踩的坑：** 你可能会问「那我直接 `var fib func(n int) int` 然后忘了赋值，调用 `fib(7)` 会怎样？」答案是运行时 panic：`runtime error: invalid memory address or nil pointer dereference`，因为 `fib` 的值还是 `nil`，调用 `nil` 函数会爆炸。所以 `var` 声明和赋值这两步缺一不可。

### ④ 斐波那契展开：朴素实现的指数级灾难与 memoization 的解法

斐波那契数列定义简单：`fib(0)=0, fib(1)=1, fib(n)=fib(n-1)+fib(n-2)`。朴素递归写出来一眼看懂，但它藏着一个性能定时炸弹。

**展开 `fib(5)` 看看：**

```
fib(5)
├── fib(4)
│   ├── fib(3)
│   │   ├── fib(2)  ← 第1次算
│   │   └── fib(1)
│   └── fib(2)      ← 第2次算，白算了！
└── fib(3)          ← 又算一次 fib(3)，白算了！
    ├── fib(2)      ← 第3次算，还在白算！
    └── fib(1)
```

`fib(2)` 被算了 3 次，`fib(3)` 被算了 2 次。随着 n 增大，重复计算呈指数级爆炸——朴素递归计算 `fib(n)` 的时间复杂度是 O(2^n)。你算 `fib(50)` 大概要等几十秒，算 `fib(100)` 的话，宇宙热寂前都跑不完。

**解法：Memoization（记忆化）。** 思路很简单，用一个 map 把算过的结果缓存起来，再次遇到同样的 n 直接查表，不重复计算。时间复杂度从 O(2^n) 瞬间降到 O(n)。生活类比：就像你第一次查了某道题的答案，把它抄在草稿纸上，下次再遇到直接看草稿，而不是重新翻书。

示意写法（不展开全部代码，意思到位）：

```go
memo := map[int]int{}
var fib func(n int) int
fib = func(n int) int {
    if n < 2 { return n }
    if v, ok := memo[n]; ok { return v }
    result := fib(n-1) + fib(n-2)
    memo[n] = result
    return result
}
```

**容易踩的坑：** 你可能觉得「我用循环+数组也能算斐波那契，何必搞递归？」完全正确！动态规划的迭代写法在工程里往往更受欢迎，因为没有递归栈的额外开销，代码也更直白。递归之所以要学，是因为它是一种**思维方式**：把大问题拆成同构小问题，这个思维在树的遍历、图的搜索、分治算法里到处都是，你绕不开它。所以，阶乘和斐波那契只是入门热身，真正的递归战场在更复杂的数据结构里等着你。

## 💡 师兄的碎碎念

- Go 的 goroutine 栈初始只有几 KB，但运行时会**自动扩容**，所以递归比 C/Java 不那么容易一下子就爆栈，不过这不是免死金牌——真写个几十万层的递归还是照样 OOM，悠着点。
- Go 编译器**不做尾递归优化**，别学某些 Scheme 教程那套「写成尾递归就等于循环」的思路；在 Go 里每一层递归调用都老老实实压一个栈帧，想省栈帧就自己手动改成迭代。
- 闭包递归有个经典坑：一定要先 `var fib func(int) int`，再用 `=` 赋值函数体，顺序不能反、符号不能用 `:=`；因为 `:=` 是边声明边赋值，右侧表达式在编译期还没见过左侧的 `fib`，直接报 undefined，这是作用域规则，不是 bug。
- 朴素递归 fib 的时间复杂度是 O(2^n)，fib(50) 就已经要算几千亿次了，实际项目里要么改成循环迭代 O(n)，要么加个 map 做 memoization（记忆化搜索），千万别把这版代码直接搬进生产环境。

## 🎓 这一关的知识点清单

- **递归**：函数在自身体内直接或间接调用自身，是解决「把大问题拆成同结构小问题」的经典范式。
- **基准条件（Base Case）**：递归必须有一个终止条件，`fact` 里是 `n == 0 return 1`，否则会无限下探直到栈溢出。
- **函数自调用**：具名函数可在函数体内直接写函数名来调用自身，Go 编译器能正确解析作用域。
- **闭包递归**：匿名函数要递归调用自身，必须先 `var fib func(int) int` 声明，再用 `=` 赋值；若用 `:=`，右侧赋值表达式在编译期还看不到左侧的 `fib`，会报「undefined」错误。
- **尾递归**：指递归调用是函数最后一个操作、返回值直接透传，理论上可被优化为循环；但 Go 编译器目前**不做**此优化，了解概念即可，不要依赖它节省栈帧。
