# 第 77 关：执行进程（师兄带你学 Go）

## 🎯 这一关你会学到

- `exec.Command` 是派生子进程，而 `syscall.Exec` 是直接把当前进程换成新程序，PID 一点没变。
- 用 `exec.LookPath` 在系统的 `PATH` 里搜索可执行文件，拿到完整的绝对路径。
- 传给 `Exec` 的参数切片，第 0 项必须是程序名本身，真正的参数从第 1 项开始。
- 用 `os.Environ()` 把当前进程的环境变量整包端过去，新程序才能看到一样的环境。
- `Exec` 一旦成功，后面的 Go 代码一行也跑不到，连 `defer` 都会被吃掉。
- 典型使用场景就是 `sudo`、`env` 这类「包装器」命令，做完准备工作就把舞台交出去。

## 🤔 先想一个问题

你有没有在终端里试过 `exec vim`？一旦敲下去，vim 退出之后整个终端 session 也跟着没了——因为 shell 进程本身已经被 vim「附体」了，vim 死了壳子自然也没了。`sudo` 也是同样的把戏：它做完权限校验，转手就把自己「变成」目标命令的进程，而不是再多生一个孩子。这一关讲的就是这个「换脸」操作，在 Go 里怎么用 `syscall.Exec` 来实现它。

## 📖 看代码

```go
// 在前面的例子中，我们了解了[生成外部进程](spawning-processes)的知识，
// 当我们需要在运行的 Go 流程中访问的外部流程时，便可以执行此操作。
// 但是有时候，我们只想用其它（也许是非 Go）的进程，来完全替代当前的 Go 进程。
// 这时，我们可以使用经典的 <a href="http://en.wikipedia.org/wiki/Exec_(operating_system)"><code>exec</code></a> 函数的 Go 的实现。

package main

import (
	"os"
	"os/exec"
	"syscall"
)

func main() {

	// 在这个例子中，我们将执行 `ls` 命令。
	// Go 要求我们提供想要执行的可执行文件的绝对路径，
	// 所以我们将使用 `exec.LookPath` 找到它（应该是 `/bin/ls`）。
	binary, lookErr := exec.LookPath("ls")
	if lookErr != nil {
		panic(lookErr)
	}

	// `Exec` 需要的参数是切片的形式的（不是放在一起的一个大字符串）。
	// 我们给 `ls` 一些基本的参数。注意，第一个参数需要是程序名。
	args := []string{"ls", "-a", "-l", "-h"}

	// `Exec` 同样需要使用[环境变量](environment-variables)。
	// 这里我们仅提供当前的环境变量。
	env := os.Environ()

	// 这里是真正的 `syscall.Exec` 调用。
	// 如果这个调用成功，那么我们的进程将在这里结束，并被 `/bin/ls -a -l -h` 进程代替。
	// 如果存在错误，那么我们将会得到一个返回值。
	execErr := syscall.Exec(binary, args, env)
	if execErr != nil {
		panic(execErr)
	}
}
```

## 🔍 师兄给你逐行拆

### Exec 不是派生是换脸

师弟，你刚学 Go 的时候，大概已经用过 `exec.Command` 启动外部程序了，感觉挺顺手——`Start()` 一调，子进程就跑起来了，父进程继续干自己的活。但这一关的主角 `syscall.Exec` 完全不是这回事，两者根本就不在一个频道上。

`exec.Command` 背后走的是操作系统的 `fork` 系统调用：内核先把当前进程完整复制一份，复制出来的那个孩子再调 `exec` 系列函数把自己的程序镜像替换成新程序。父进程全程没有消失，父子两个进程并行存在，父进程还能 `Wait()` 等孩子死透再收尸。这就是经典的「fork + exec」二段式，教科书上的标准答案。

但 `syscall.Exec` 只做第二段，而且是在当前进程上直接做。它调的是 Linux 的 `execve` 系统调用，内核会把当前进程的代码段、数据段、堆栈全部丢掉，把新程序的镜像塞进来，然后从新程序的入口点跑起来——但 PID 没变，打开的文件描述符默认也继承下来了，父进程眼里这个 PID 还在，只是突然「灵魂换了个人」。

用角色扮演剧打比方吧：`exec.Command` 是导演喊「NG，换个替身来跑这场戏」，替身是个新演员，原来那个演员还站在旁边看；而 `syscall.Exec` 是「不换人，直接给这个演员戴上新角色的面具、换上新角色的剧本、注射新角色的记忆」，角色名（PID）没变，但演的东西完全不同了。这个区别听起来玄，但实际编程里影响非常大，后面你会体会到。

### LookPath 帮你找到绝对路径

你可能会想，我直接传个 `"ls"` 给 `syscall.Exec` 不就行了吗？师兄我当年也这么想过，然后被操作系统狠狠教育了一顿。`execve` 这个系统调用要求的是可执行文件的**绝对路径**，它不认识「相对路径」或者「裸命令名」，你传一个 `"ls"` 它根本不知道去哪里找。

`exec.LookPath` 就是干这个的：它接受一个命令名，然后按照 `PATH` 环境变量里列出的目录，一个一个去找这个名字的可执行文件，找到了就把完整的绝对路径还给你，比如 `/bin/ls`。如果找不到，它会返回一个错误，你得老老实实处理掉。

类比一下：你让快递员给你送个外卖，你只说「就对面那栋楼」，快递员肯定懵——他需要的是「XX 市 XX 区 XX 路 88 号 3 楼 302」这种完整地址。`LookPath` 就是那个帮你把「对面那栋楼」翻译成完整门牌号的翻译官，没有它你的包裹（进程）就别想发出去。

顺带说一句，`LookPath` 返回的路径还有个好处：它确认了这个文件真实存在并且有可执行权限，省得你传进去一个根本没有 `+x` 位的文件，到 `Exec` 那一步才报错，那就尴尬了。所以这个函数虽然看起来就一行，但一定不要省掉，该写就写，防御性编程嘛。

### args 和 env 两个细节

`syscall.Exec` 的函数签名大概是「路径、参数切片、环境变量切片」三个参数，听起来简单，但里面有两个细节师弟你一定得记牢，不然踩坑了找不到原因会很崩溃。

**第一个细节：args 切片的第 0 项是程序名本身。** 这个约定来自 C 语言的 `main(int argc, char *argv[])` 传统——`argv[0]` 按惯例放的是程序自己的名字，`argv[1]` 开始才是真正的命令行参数。Unix 世界里有不少工具会读 `argv[0]` 来决定自己的行为，比如 `busybox` 这个神器，它一个可执行文件内置了几百个小工具，靠的就是运行时读 `argv[0]` 看自己叫什么名字，叫 `ls` 就走 `ls` 的逻辑，叫 `grep` 就走 `grep` 的逻辑。所以在 Go 里构造 args 的时候，通常写成 `[]string{"ls", "-a", "-l"}` 这样，第 0 项填程序名，后面跟参数，这是正统写法，别偷懒漏掉第 0 项。

**第二个细节：env 用 `os.Environ()` 把当前环境整包传过去。** `os.Environ()` 返回的是形如 `["HOME=/root", "PATH=/usr/bin:/bin", ...]` 的字符串切片，就是当前进程所有环境变量的快照。你要是不传或者传一个空切片，新程序启动后环境变量一片空白，它找不到 `HOME`、找不到 `PATH`、配置也全丢了，很可能直接挂掉或者行为异常。当然你也可以自己构造一个精简的 env 切片，只给新程序喂它需要的那几个变量，这样更干净，用于安全场景的时候尤其有用。但日常包装器场景里，直接 `os.Environ()` 图个省心就行了。

这两个细节加在一起，写出来的调用大概就是：路径用 `LookPath` 拿到的结果，args 第 0 项是程序名、后面跟参数，env 用 `os.Environ()`。三件套缺一不可，记住了吗？

### 成功就再见，失败才返回

最后这个特性是 `syscall.Exec` 最反直觉的地方，也是最容易让新人懵圈的地方：**`Exec` 调用成功之后，这行代码后面的所有 Go 代码永远不会被执行到。** 不是「可能不会」，是「永远不会」，包括你精心写的 `defer`、包括后面的错误处理、包括 `fmt.Println("done")` 这种调试打印，全部都跑不到。

道理很简单：`Exec` 成功了，当前进程的 Go 运行时已经被内核抹掉了，换成了新程序的镜像，Go 的调用栈不存在了，`defer` 注册的清理函数自然也没地方跑了。这就像你把手机恢复出厂设置，恢复完之后原来装的 App 当然也没了，没有什么神奇的力量能让它们在出厂系统上继续跑。

正因为如此，`syscall.Exec` 的返回值只有一个 `error`，而且**只有失败的时候才会返回**。如果 `execErr` 不是 `nil`，说明 `Exec` 没有成功启动新程序（比如路径错了、权限不够），这时候当前进程还活着，你得处理这个错误。如果 `execErr` 是 `nil`，额，你根本不会走到判断它的那行代码了——因为进程已经「换脸」成功了。

这个特性决定了 `syscall.Exec` 的使用姿势：你要在调用它之前把所有该做的准备工作做完，文件该关的关，日志该写的写，然后调 `Exec`，把舞台彻底交出去，自己就此功成身退。`sudo` 就是这么干的：验证密码、设置权限、准备好环境，最后一句 `Exec` 把自己变成目标命令，既优雅又干净，没有多余的进程在那里占坑。师弟你以后写类似的命令行工具，也可以学这个范式，用 `Exec` 做最后的「禅让」动作，特别帅。

## 🏃 跑一下试试

在 macOS 或 Linux 下，直接使用 `go run` 运行本示例即可观察到效果。Go 运行时会先通过 `exec.LookPath("ls")` 定位到 `ls` 的绝对路径（通常是 `/bin/ls`），随后调用 `syscall.Exec` 将当前 Go 进程**原地替换**为 `ls` 进程，并传入 `-a`、`-l`、`-h` 参数与完整的环境变量。整个过程中进程 PID 保持不变，Go 运行时彻底消失，最终你看到的输出完全由 `ls` 产生。

```shell
go run execing-processes.go
```

典型输出大致如下（内容因目录而异）：

```text
total 48
drwxr-xr-x  4 alice staff  128B  6 10 10:22 .
drwxr-xr-x 12 alice staff  384B  6 10 09:55 ..
-rw-r--r--  1 alice staff  1.2K  6 10 10:22 execing-processes.go
```

请注意：**Windows 平台不支持 `syscall.Exec`**，该函数在 Windows 上是一个空桩，调用后会立即返回 `EWINDOWS` 错误，程序不会崩溃但也不会执行任何替换操作。如果需要跨平台兼容，应当改用 `os/exec.Command` 来启动子进程。

## 💡 师兄的碎碎念

师兄碎碎念时间到。

**第一件事：Windows 上的 `syscall.Exec` 是空桩。** Go 的标准库为了让代码能在 Windows 上编译通过，给 `syscall.Exec` 提供了一个什么都不做、直接返回错误的实现。这意味着你的程序在 Linux/macOS 上跑得飞起，一到 Windows 就悄悄失效。真正的跨平台做法是用构建标签（`//go:build`）进行条件编译：Unix 走 `syscall.Exec`，Windows 走 `exec.Command` 启动子进程。两者语义有差异，需要业务层自己兜底。

**第二件事：`defer` 全部作废。** 这一点非常容易踩坑。`syscall.Exec` 一旦成功，操作系统内核会直接把当前进程的代码段、堆栈全部替换掉，Go 运行时根本没有机会执行任何清理逻辑。换句话说，你在 `Exec` 之前注册的所有 `defer`——关闭文件、释放锁、刷新缓冲区——统统不会跑。所以在调用 `Exec` 之前，务必手动完成所有必要的收尾工作。

**第三件事：Unix 的 fork + exec 模型。** 我们平时用的 `sudo`、`nohup`，以及 Shell 内建命令 `exec`，背后走的都是同一套操作系统原语。Unix 的经典做法是：父进程调用 `fork()` 复制自身得到子进程，子进程再调用 `exec()` 家族函数加载新程序。Go 的 `syscall.Exec` 省掉了 fork，直接在当前进程原地换血，是一种更激进但也更高效的用法。理解这个模型，对排查僵尸进程、信号传递等问题大有裨益。

- [ ] 明确 `os.Environ()` 的用途：将当前进程的环境变量完整继承给新进程。
