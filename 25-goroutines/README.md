# 第 25 关：协程（师兄带你学 Go）

## 🎯 这一关你会学到

本关搞懂 Go 的协程（goroutine）：只需在函数调用前加上 `go` 关键字，就能让这个函数在独立的并发流中执行。goroutine 极其轻量，初始栈仅几 KB，可按需扩缩。成千上万个 goroutine 同时跑，Go 运行时帮你调度，完全无压力。

## 🤔 先想一个问题

想象一家忙碌的奶茶店——你刚下单，后厨立刻分工：一个店员专心煮茶底，一个店员麻利地切水果，一个店员负责装杯封口，三条流水线互不干扰、同步推进，出单速度远超一个人从头做到尾。"店员"——写一个函数，前面加个 `go`，它就被扔到另一条流水线跑，`main` 自己不等它、继续往下做。这是 Go 最吸引人的并发模型——启动协程的成本几乎可以忽略不计。

**goroutine 是 Go 运行时管理的轻量级线程，用 `go f()` 语法启动；它比 OS 线程便宜得多（栈仅 2KB 起，可动态扩），一个 Go 程序跑 10 万个协程毫无压力。**

## 📖 看代码

```go
// _协程(goroutine)_ 是轻量级的执行线程。

package main

import (
	"fmt"
	"time"
)

func f(from string) {
	for i := 0; i < 3; i++ {
		fmt.Println(from, ":", i)
	}
}

func main() {

	// 假设我们有一个函数叫做 `f(s)`。
	// 我们一般会这样 `同步地` 调用它
	f("direct")

	// 使用 `go f(s)` 在一个协程中调用这个函数。
	// 这个新的 Go 协程将会 `并发地` 执行这个函数。
	go f("goroutine")

	// 你也可以为匿名函数启动一个协程。
	go func(msg string) {
		fmt.Println(msg)
	}("going")

	// 现在两个协程在独立的协程中 `异步地` 运行，
	// 然后等待两个协程完成（更好的方法是使用 [WaitGroup](waitgroups)）。
	time.Sleep(time.Second)
	fmt.Println("done")
}
```

## 🔍 师兄给你逐行拆

### 同步调用：f("direct") 老老实实等它跑完

```go
f("direct")
```

这就是最普通的同步函数调用。`main` 调用 `f` 之后，**必须等 `f` 里三行打印全部跑完**，才会继续往下执行。输出结果是：

```
direct : 0
direct : 1
direct : 2
```

顺序严格按照调用顺序，一行接一行，绝不乱序。这是典型的**单线程思维：一个一个排队做**，当前任务没完成，后续代码纹丝不动。

生活类比：你去奶茶店**自己排队、自己点单、自己付钱**——一条龙全靠自己搞定。排队没完就不能点单，点单没完就不能付钱，每一步都在等上一步，稳稳当当但也慢慢吞吞。同步调用的好处是**逻辑清晰、结果可预期**，坏处是一旦某个函数耗时很长，整个程序就只能干等着。

### go f()：一行就起一个协程

```go
go f("goroutine")
```

在函数调用前加一个 `go` 关键字，Go runtime 就会把 `f` 丢到一条独立的执行线里跑——这就是 **goroutine（协程）**。`main` 函数**不会等它**，而是立刻继续往下执行。

所以当你同时跑 `f("goroutine")` 和主流程里的打印语句，终端输出的顺序是**不固定的**：`"goroutine"` 和 `"going"` 会交错出现，每次运行结果可能不同。

Go runtime 在背后自动把所有 goroutine **调度到 OS 线程**上执行，你完全不用手动管理线程。

语法就这么简单——原本的函数调用一字不改，**只在最前面加两个字母 `go`**，并发就启动了。这也是 Go 并发模型最吸引人的地方：门槛极低，表达却很直接。

### go + 匿名函数：最轻的"丢任务"姿势

```go
go func(msg string) {
    fmt.Println(msg)
}("going")
```

这是 Go 并发的**万能姿势**：把一个匿名函数直接作为协程启动，立即给它传参。适合"写到哪想起要并发就现场开一个"的场合，不用专门声明一个命名函数。

**要小心闭包陷阱**：匿名函数里引用外层循环变量时要特别注意。Go 1.22 之前，`for i := range ...` 的 `i` 在整个循环里共享，所有协程捕获的都是同一个 `i`，往往看到的都是最后一个值。惯用法是把变量当参数传进去：

```go
for _, v := range items {
    go func(v T){ process(v) }(v)  // 传参隔离
}
```

Go 1.22+ 每轮循环会重新声明变量，这个坑变小了但别依赖——老代码里还是要留心。

### 等协程跑完：time.Sleep 是权宜之计，正道是 channel/WaitGroup

前面示例里的 `time.Sleep(time.Second)` 只是让主协程"撑"够一秒，给子协程留时间——**生产代码绝不能这样写**：睡多久是拍脑袋，协程快了浪费时间，慢了照样被砍。

**核心陷阱**：`main` 函数一旦返回，整个进程立即结束，所有仍在运行的协程不论做没做完，全部强制终止。因此必须显式同步等待。

正规做法有两种：

① **`sync.WaitGroup`**：启动前 `wg.Add(1)`，协程内 `defer wg.Done()`，主协程调 `wg.Wait()` 阻塞直到计数归零。

```go
var wg sync.WaitGroup
wg.Add(1)
go func() {
    defer wg.Done()
    fmt.Println("working")
}()
wg.Wait()
```

② **channel 信号**：协程完成后向 channel 发值，主协程读取即解除阻塞。适合需要传递结果或精细控制的场景。

记住：`time.Sleep` 等协程是**定时炸弹**，`WaitGroup` 才是正道。

## 💡 师兄的碎碎念

- goroutine 初始栈仅 **2 KB**，运行时按需动态扩缩，最大可达 GB 级，远比 OS 线程（默认 1–8 MB）轻量
- 同时跑 **10 万个** goroutine，内存消耗通常只有几百 MB，高并发场景无需担心资源耗尽
- **channel 是 goroutine 间通信的首选**，遵循「不要通过共享内存通信，而要通过通信共享内存」原则，避免用共享变量 + 锁
- `time.Sleep` 等待协程是**反模式**：时间难以估算且不可靠，应改用 `sync.WaitGroup` 或 channel 做确定性同步

## 🎓 这一关的知识点清单

- **goroutine 概念**：Go 运行时管理的轻量级执行单元，初始栈 2KB，非 OS 线程，由 Go 调度器（GMP 模型）映射到 OS 线程执行。
- **`go` 关键字语法**：在函数调用前加 `go`，即可把它启动为独立协程，调用方不等它返回继续执行。
- **并发非确定性**：多个协程的执行顺序由运行时调度决定，每次运行输出顺序可能不同，不能依赖。
- **等待机制**：用 `sync.WaitGroup` 或 channel 阻塞等待协程结束，禁止用 `time.Sleep` 估时间。
- **协程 vs OS 线程**：协程在用户态调度成本极低，OS 线程由内核调度成本较高；Go 可以轻松跑 10 万+ 协程，但同等规模 OS 线程会内存爆炸。
