# 第 27 关：通道缓冲（师兄带你学 Go）

## 🎯 这一关你会学到

本关学习带缓冲 channel 的创建与使用。通过 `make(chan T, N)` 可以创建一个容量为 N 的带缓冲 channel。与无缓冲 channel 不同，带缓冲 channel 允许发送方在 channel 未满时直接将值放入，而无需等待接收方就绪，从而避免了不必要的阻塞，提升了并发程序的灵活性与吞吐能力。

## 🤔 先想一个问题

想象一下小区里的快递柜。无缓冲 channel 就像快递员必须亲手把包裹交到你手上——你不在家，快递员就只能站在门口干等；你不接，他就走不了。双方必须同时在场，时间完全耦合。带缓冲 channel 则像一排快递柜——快递员把包裹塞进格子、关上柜门就可以离开去跑下一单，完全不用管你什么时候回来；你下班回家，有空了再去扫码取件。只要柜子没塞满，快递员永远不会被卡住。"快递柜"就是带缓冲的 channel——在容量范围内，收发解耦，各自自由。

**带缓冲 channel 内部有一块 FIFO 队列，容量 N 表示最多能暂存 N 个未被接收的值。发送在队列未满时不阻塞，接收在队列非空时不阻塞；队满或队空都按无缓冲规则阻塞。**

## 📖 看代码

```go
// 默认情况下，通道是 _无缓冲_ 的，这意味着只有对应的接收（`<- chan`）
// 通道准备好接收时，才允许进行发送（`chan <-`）。
// _有缓冲通道_ 允许在没有对应接收者的情况下，缓存一定数量的值。

package main

import "fmt"

func main() {

	// 这里我们 `make` 了一个字符串通道，最多允许缓存 2 个值。
	messages := make(chan string, 2)

	// 由于此通道是有缓冲的，
	// 因此我们可以将这些值发送到通道中，而无需并发的接收。
	messages <- "buffered"
	messages <- "channel"

	// 然后我们可以正常接收这两个值。
	fmt.Println(<-messages)
	fmt.Println(<-messages)
}
```

## 🔍 师兄给你逐行拆

### make(chan T, N)：加个数字就有缓冲

```go
messages := make(chan string, 2)
```

第二个参数 `N=2` 是**缓冲大小**，内部维护一个容量为 2 的 FIFO 队列。

**无缓冲** `make(chan string)`（等价于 `N=0`）要求发送方和接收方必须同时就绪，像两人面对面交接——缺一不可。

**有缓冲**则不同：队列没满时，发送方可以直接把值塞进去然后继续跑，不必等接收方到场；队列非空时，接收方也可以随时取走，不必等发送方再次发送。

```go
messages <- "ping"   // 不阻塞，塞入队列
messages <- "pong"   // 队列满了，再发就会阻塞

fmt.Println(cap(messages)) // 2，容量固定不变
fmt.Println(len(messages)) // 2，当前队列里有 2 个元素
```

`cap(ch)` 返回创建时指定的容量，`len(ch)` 返回队列中**当前**积压的元素数。"水位"，判断是否接近满。

### 缓冲未满：连续发两个也不阻塞

```go
messages <- "buffered"
messages <- "channel"
```

这两行连续向同一个 channel 发送数据。由于我们创建的 channel 缓冲容量为 2，而此时队列还是空的，两次发送都**立刻返回**，不会阻塞——发送方根本不需要等接收方出现。

这正是缓冲 channel 的核心价值：**解耦发送方和接收方的时间**。生产者可以先攒一批数据扔进队列，消费者有空再慢慢取，双方不必卡在同一时刻握手。

不过缓冲是有上限的。如果此时再执行第三次 `messages <- "third"`，队列已满，发送操作就会**阻塞**，直到有接收方取走一条数据腾出空位为止。缓冲不是无底洞，只是给双方多了一点喘息空间。

### 按 FIFO 接收：先进先出

```go
fmt.Println(<-messages) // "buffered"
fmt.Println(<-messages) // "channel"
```

Go 的 channel 内部是一个环形队列，元素严格按**先进先出**顺序弹出。第一次接收取出 `"buffered"`，第二次取出 `"channel"`，顺序与写入完全一致。

队列取空后，再次执行 `<-messages` 会**阻塞**，直到有新值被发送进来；若 channel 已被 `close()`，则立即返回该类型的零值（以及 `false` 的 ok 标志）。

---

**该选多大的缓冲？"延后问题"：如果消费速度长期跟不上生产，再大的缓冲也会爆。缓冲主要用于**平滑瞬时峰值**、**时间解耦**，不是异步化的捷径。

### 陷阱：缓冲不是异步的万能药

很多初学者认为给 channel 加上缓冲就能一劳永逸地避免死锁，这是一个危险的误区。"先塞进去不等"，但如果消费者根本不来取，队列塞满后发送方照样阻塞，死锁依旧。

**缓冲大小拍脑袋**：随手写 100、1000，可能掩盖了"消费跟不上生产"的真实瓶颈，出事时反而更难排查。

**经验做法**：默认用无缓冲；只有在明确证据表明需要解耦时机，并且能算出合理的容量上限时，再开缓冲。缓冲不是性能优化工具，是**时间解耦**工具——它让双方不必同时到场，仅此而已。

```bash
$ go run channel-buffering.go
buffered
```

两次发送都在同一个主协程里完成、没开子协程，因为缓冲容量 2 足够容纳两条消息；之后按 FIFO 顺序依次读出，先发的 `"buffered"` 先被接收到。

## 💡 师兄的碎碎念

- `cap(ch)` 返回 channel 的固定缓冲容量，`len(ch)` 返回当前队列中待读取的元素数；两者结合可判断 channel 的「水位」
- 向**已满**的缓冲 channel 发送数据，或从**空**的缓冲 channel 接收数据，协程都会阻塞，直到对端完成对应操作
- 默认应优先使用**无缓冲** channel——它能强制发送方与接收方在时间上同步，逻辑更清晰、更易推理
- 缓冲 channel 可充当**信号量**：`sem := make(chan struct{}, N)` 在进入临界区前发送、退出后接收，即可将并发数限制在 N 以内

## 🎓 这一关的知识点清单

- **带缓冲 channel 语法**：`make(chan T, N)`，第二个参数 N 是缓冲容量。
- **FIFO 队列**：内部环形队列，严格先进先出；`close` 后仍可继续读完剩余元素。
- **cap vs len**：`cap(ch)` 是创建时指定的容量（固定不变），`len(ch)` 是当前队列中积压的元素数。
- **满/空阻塞规则**：向满 channel 发送阻塞到有空位；从空 channel 接收阻塞到有值送来（或 channel 关闭后返回零值）。
- **选择缓冲大小的原则**：默认用无缓冲；缓冲只用于解耦时间（不是提速）；不要拿超大缓冲掩盖消费速度不足。
