# 第 17 关：指针（师兄带你学 Go）

## 🎯 这一关你会学到

这一关你会搞清楚一件事：把变量传给函数时，到底是「复印一份给它」还是「把原件地址给它」。Go 里用 `&` 可以拿到变量在内存里的地址，用 `*` 可以顺着地址找到原来的值并修改它——这俩符号就是指针的全部秘密。简单说，指针就是一个专门存内存地址的变量，学会它你就能在函数里真正改掉外面的数据，而不是白忙一场。

## 🤔 先想一个问题

你饿了，让室友帮你拿外卖。有两种方式：第一种，把外卖的截图发给他——他拿到的是一份**副本**，看得到信息，却改不了原单；第二种，把外卖柜的**取件码**发给他——他直接找到那个柜子，取出原件，甚至能替你签收、修改备注。截图再清晰，也只是复制品；取件码才真正指向那个柜子本身。Go 里的函数传参默认走第一条路，值被复制一份传进去，函数内的修改不会影响外面的原变量。而指针走的是第二条路，它存的不是数据本身，而是数据住在哪里——有了地址，就能找到原件，就能直接修改它。这就是指针存在的意义：节省复制开销，并且让函数真正触碰到原始数据。

**指针是存放另一个变量内存地址的变量。Go 里 `*T` 是指向 T 的指针类型，`&变量` 取地址，`*指针` 访问指向的值。**

## 📖 看代码

```go
// Go 支持 <em><a href="http://zh.wikipedia.org/wiki/%E6%8C%87%E6%A8%99_(%E9%9B%BB%E8%85%A6%E7%A7%91%E5%AD%B8)">指针</a></em>，
// 允许在程序中通过 `引用传递` 来传递值和数据结构。

package main

import "fmt"

// 我们将通过两个函数：`zeroval` 和 `zeroptr` 来比较 `指针` 和 `值`。
// `zeroval` 有一个 `int` 型参数，所以使用值传递。
// `zeroval` 将从调用它的那个函数中得到一个实参的拷贝：ival。
func zeroval(ival int) {
	ival = 0
}

// `zeroptr` 有一个和上面不同的参数：`*int`，这意味着它使用了 `int` 指针。
// 紧接着，函数体内的 `*iptr` 会 _解引用_ 这个指针，从它的内存地址得到这个地址当前对应的值。
// 对解引用的指针赋值，会改变这个指针引用的真实地址的值。
func zeroptr(iptr *int) {
	*iptr = 0
}

func main() {
	i := 1
	fmt.Println("initial:", i)

	zeroval(i)
	fmt.Println("zeroval:", i)

	// 通过 `&i` 语法来取得 `i` 的内存地址，即指向 `i` 的指针。
	zeroptr(&i)
	fmt.Println("zeroptr:", i)

	// 指针也是可以被打印的。
	fmt.Println("pointer:", &i)
}
```

## 🔍 师兄给你逐行拆

### 值传递：zeroval 为什么白忙一场

先看这段代码：`func zeroval(ival int) { ival = 0 }`。调用它的时候，Go 会把你传进去的变量 `i` 的**值复制一份**，塞到一个叫 `ival` 的新变量里。这两个变量住在不同的内存地址——`i` 住在 `main` 函数的栈帧上，`ival` 住在 `zeroval` 自己的栈帧上，互不相识。

所以函数里写 `ival = 0`，改的只是那个副本。等 `zeroval` 执行完毕、栈帧一销毁，`ival` 灰飞烟灭，原来的 `i` 压根儿没被碰过，依然是原来那个值——白忙一场，毫无意义。

**生活类比**：你把外卖拍了张照片发给室友，室友拿马克笔在照片上乱画，你桌上那份真外卖一点也不会变。`ival` 就是那张照片，`i` 才是真外卖。

这里要敲一下新手的脑壳：很多人写 `zeroval(i)` 之后，满心期待 `i` 变成 0，结果一脸懵逼。

### & 取地址：把变量的内存门牌号交出去

内存就像一栋大楼，每个变量住在一间宿舍里，`&` 就是去门口抄下那间宿舍的**门牌号**。

```go
i := 42
p := &i   // p 拿到了 i 的门牌号
fmt.Println(p)   // 输出类似 0xc0000b4008
fmt.Println(*p)  // 顺着门牌找进去，得到 42
```

`&i` 的类型是 `*int`——意思是「一个指向 int 宿舍的门牌号」。门牌号本身只是一串数字（内存地址），它不等于里面住的人（变量的值）。

> ⚠️ **只有左值才能取地址。**
> 左值是「有固定宿舍的变量」，而 `42` 只是一个临时写在纸上的数字，没有宿舍，自然也没有门牌号。

```go
p := &42  // ❌ 编译报错：cannot take the address of 42
```

记住这条规则：**能放在赋值号左边的，才能取地址。**

### * 解引用：顺着门牌号摸回去改值

有了指针，光「记住门牌号」没用，得**推门进屋改东西**——这就是解引用，语法是在指针前面再加一个 `*`。

```go
func zeroptr(iptr *int) {
    *iptr = 0  // 顺着门牌号找到房间，把里面的值改成 0
}

func main() {
    i := 42
    zeroptr(&i)  // 把 i 的「门牌号」递给师兄
    fmt.Println(i) // 输出 0
}
```

`&i` 是「把门牌号抄给你」，`*iptr` 是「按号推门进屋」。师兄在函数里操作的不是副本，而是你家那个真实的 `i`，所以调完之后 `i` 就真的变 0 了。

> **踩坑预警：nil 指针解引用会直接 panic**

```go
var p *int  // p 是 nil，相当于一张空白门牌，根本没有对应的房间
*p = 1      // 按图索骥却发现门牌是假的——程序直接崩掉
```

`nil` 指针就像一张写着「无效地址」的门牌，Go 运行时一看你要推门，立刻抛出 `panic: runtime error: invalid memory address or nil pointer dereference`。用指针前先判断是否为 `nil`，是保命的好习惯。

### 打印指针：那串 0x4213... 是啥

执行 `fmt.Println("pointer:", &i)` 时，终端会打出类似 `0xc0000b4008` 的十六进制数——这就是变量 `i` 在内存中的地址。每次运行程序，操作系统分配的地址都不同，所以这串数字会变来变去，属于正常现象。

这里顺带提两个 Go 指针的关键特性：

**① 指针不能做算术运算。** C/C++ 里可以写 `p + 1` 来跳到下一个元素，Go 直接禁止这种操作，编译器会报错。非要搞指针偏移，必须引入 `unsafe` 包——名字已经暗示你：后果自负。这个限制让 Go 规避了一大类内存越界 bug。

**② 逃逸分析保你安全。** 在 C 里，返回局部变量的地址是未定义行为，函数栈帧销毁后那块内存就是野指针。

$ go run pointers.go

注意 `pointer:` 后面那串十六进制地址是程序运行时分配的，你在自己机器上跑出来的数字一定和上面不一样，属于正常现象。

## 💡 师兄的碎碎念

- **nil 指针**：解引用 nil 指针会直接 panic，没有任何征兆，用之前养成习惯先 `if p != nil` 判一下，救你于水火。
- **逃逸分析**：不用担心「返回了局部变量的地址会悬空」，Go 编译器有逃逸分析，它会悄悄把该变量挪到堆上，你大胆返回就好。
- **不能指针算术**：Go 故意砍掉了 `p+1` 这种玩法，想搞底层偏移必须用 `unsafe.Pointer`，但那是开了安全锁的区域，没事别乱碰。
- **传指针 vs 传值**：大结构体传指针省拷贝没错，但小值类型（int、小 struct）传值反而更缓存友好，别迷信「指针一定快」。

## 🎓 这一关的知识点清单

- **指针**：存储变量内存地址的特殊变量，类型写作 `*T`，通过它可间接访问或修改原始数据。
- **取地址符 `&`**：放在变量前可获取该变量的内存地址，返回一个指向该变量的指针，如 `p := &x`。
- **解引用 `*`**：放在指针变量前可访问或修改指针所指向的实际值，如 `*p = 42` 会直接改变原变量。
- **值传递与引用语义**：Go 函数参数默认为值传递（拷贝副本），传递指针可模拟引用语义，使函数内部的修改作用于原始变量。
- **nil 零值**：指针类型的零值为 `nil`，表示不指向任何地址；解引用 `nil` 指针会引发运行时 panic，使用前须做非空检查。
