# 第 70 关：命令行标志（师兄带你学 Go）

## 🎯 这一关你会学到

- `flag` 包是 Go 标准库里专门用来解析命令行参数（比如 `-n 10`、`-v`）的工具，不用自己手动拆 `os.Args`。
- `flag.String`、`flag.Int`、`flag.Bool` 这些函数返回的是**指针**，读值时要在前面加 `*` 解引用，否则拿到的只是地址。
- 如果你手头已经有一个变量，可以用 `flag.StringVar`、`flag.IntVar` 等把它直接绑定到某个标志，省去解引用那一步。
- 所有标志声明完之后，**必须调用一次 `flag.Parse()`**，不调用的话标志永远是默认值，程序读不到你传进去的内容。
- 解析完标志剩下的"位置参数"可以用 `flag.Args()` 拿到，像 `git commit main.go` 里的 `main.go` 就是这种。

## 🤔 先想一个问题

你平时敲 `wc -l file.txt`、`ls -a`、`curl -X POST` 的时候，有没有想过那些横杠参数是怎么来的？如果你自己写了一个 Go 小工具，想让别人跑的时候传 `-name Alice` 或者 `-port 8080`，难道要自己去切割 `os.Args` 一个字一个字地比对吗？其实根本不用那么麻烦，Go 标准库的 `flag` 包帮你把这些脏活全包了，这一关我们就来看看怎么用它优雅地接收命令行标志。

## 📖 看代码

```go
// [_命令行标志_](http://en.wikipedia.org/wiki/Command-line_interface#Command-line_option)
// 是命令行程序指定选项的常用方式。例如，在 `wc -l` 中，
// 这个 `-l` 就是一个命令行标志。

package main

// Go 提供了一个 `flag` 包，支持基本的命令行标志解析。
// 我们将用这个包来实现我们的命令行程序示例。
import (
	"flag"
	"fmt"
)

func main() {

	// 基本的标记声明仅支持字符串、整数和布尔值选项。
	// 这里我们声明一个默认值为 `"foo"` 的字符串标志 `word` 并带有一个简短的描述。
	// 这里的 `flag.String` 函数返回一个字符串指针（不是一个字符串值），
	// 在下面我们会看到是如何使用这个指针的。
	wordPtr := flag.String("word", "foo", "a string")

	// 使用和声明 `word` 标志相同的方法来声明 `numb` 和 `fork` 标志。
	numbPtr := flag.Int("numb", 42, "an int")
	forkPtr := flag.Bool("fork", false, "a bool")

	// 用程序中已有的参数来声明一个标志也是可以的。
	// 注意在标志声明函数中需要使用该参数的指针。
	var svar string
	flag.StringVar(&svar, "svar", "bar", "a string var")

	// 所有标志都声明完成以后，调用 `flag.Parse()` 来执行命令行解析。
	flag.Parse()

	// 这里我们将仅输出解析的选项以及后面的位置参数。
	// 注意，我们需要使用类似 `*wordPtr` 这样的语法来对指针解引用，
	// 从而得到选项真正的值。
	fmt.Println("word:", *wordPtr)
	fmt.Println("numb:", *numbPtr)
	fmt.Println("fork:", *forkPtr)
	fmt.Println("svar:", svar)
	fmt.Println("tail:", flag.Args())
}
```

## 🔍 师兄给你逐行拆

### 先说说 flag 包到底在干嘛

上一关我们自己扒 `os.Args`，就像去菜市场买菜——你得一个摊位一个摊位地问价、挑选、还要自己掂量斤两，全程手动，稍不注意就买错了。而 `flag` 包呢，更像是一家装修整齐的自助餐厅：你提前告诉服务员「我要一个叫 `name` 的字符串，默认值是 world」，剩下的事情——从命令行里找到这个参数、把它转换成正确的类型、在用户输入 `-help` 时自动打印说明——全都由 `flag` 包替你搞定，你只管端着盘子去取结果就行。

具体来说，`flag` 包帮你做了三件事。第一，**声明参数**：你用 `flag.String`、`flag.Int`、`flag.Bool` 这类函数提前登记好每个参数的名字、类型和默认值，相当于提前在餐台上摆好了对应的餐盘。第二，**自动解析**：只要在 `main` 函数里调用一次 `flag.Parse()`，包内部就会遍历 `os.Args`，按名字找到对应的值并完成类型转换，一行代码顶过去好几十行的手撸逻辑。第三，**免费送帮助文档**：当用户在终端输入 `-help` 或者写错了参数名，`flag` 包会自动把你声明过的所有参数连同默认值一起打印出来，体验非常丝滑，完全不需要你额外写一个 `printUsage` 函数。

总结一句话：`flag` 包就是把「读取命令行参数」这件繁琐的体力活，升级成了一套声明式的、开箱即用的标准流程，让你把精力留给真正的业务逻辑。

### 声明标志的两种姿势

Go 的 `flag` 包提供了两种声明命令行参数的方式，师兄今天给你捋清楚，省得你到时候对着指针发呆。

**第一种**是直接调用 `flag.String`、`flag.Int`、`flag.Bool` 这类函数。它们的返回值是**指针**，也就是说你拿到的不是值本身，而是一张「号码牌」。就好比食堂打饭，服务员给你一张取餐号，饭还在窗口那边，你得自己拿着号去取。代码里的体现就是：你必须在 `flag.Parse()` 之后，用星号解引用才能读到真实的值，比如 `*name`、`*port`。写惯了其他语言的同学刚开始容易忘记这个星号，结果打印出来是一串地址，看得一脸懵。

```go
name := flag.String("name", "world", "要问候的对象")
flag.Parse()
fmt.Println("Hello,", *name) // 注意解引用
```

**第二种**是 `flag.StringVar`、`flag.IntVar`、`flag.BoolVar` 这族函数。区别在于你先自己声明好变量，然后把变量的地址传进去，`flag` 包解析完成后会直接把结果写进那个变量。这就像食堂直接把菜端到你桌上，你坐在那儿等着，抬手就能吃，完全不用拿号、不用跑窗口。使用时直接读变量名就行，没有星号，代码看起来更干净。

```go
var name string
flag.StringVar(&name, "name", "world", "要问候的对象")
flag.Parse()
fmt.Println("Hello,", name) // 直接用变量名
```

两种方式在功能上完全等价，选哪种看你的习惯。如果参数较多、需要集中管理，师兄建议用第二种，把所有变量统一声明在结构体或函数顶部，逻辑更清晰，别人读代码也更容易跟上思路。总之，记住核心区别：第一种给你指针，用前要解引用；第二种替你填好变量，拿来即用。

### 三个参数：名字、默认值、帮助说明

师弟，你有没有注意到，`flag.String`、`flag.Int`、`flag.Bool` 这些函数，不管哪一个，调用时都要传三个参数？这不是巧合，而是 `flag` 包设计上的统一规范，每个参数都有自己明确的职责，缺一不可。

**第一个参数是标志名**，也就是用户在命令行里实际敲的那串字符。比如你传了 `"port"`，用户就要写 `-port 9090` 才能触发这个标志。名字要简短、直观，让人一眼就知道这个选项是干什么的。

**第二个参数是默认值**，当用户启动程序时没有传这个标志，程序就会自动使用你在这里填写的值。这个参数至关重要，一定要设得合理。举个例子，声明一个 HTTP 服务的端口标志可以这样写：

```go
port := flag.Int("port", 8080, "HTTP 服务端口号")
```

这里默认值是 `8080`，这是 Web 开发里约定俗成的常用端口，用户如果懒得传参数，程序直接用 `8080` 启动，完全没有问题。但如果你把默认值随手写成 `0` 或者干脆留空，用户一旦忘记传参，程序轻则行为异常，重则直接崩掉，到时候排查起来会非常头疼，所以默认值一定要经过认真思考再填写。

**第三个参数是帮助说明**，这是一段简短的文字描述。当用户在命令行执行 `-h` 或 `--help` 时，`flag` 包会自动把所有标志的名字、默认值和这段说明整整齐齐地打印出来，方便用户快速了解每个选项的用途。说明文字不需要长篇大论，简明扼要即可，比如上面例子里的 `"HTTP 服务端口号"` 就足够清晰了。

记住这三个参数的顺序：名字、默认值、帮助说明，以后不管声明什么类型的标志，套路都是一样的，熟练之后自然就记住了。

### 核心开关：flag.Parse 必须调用一次

小师弟，你有没有玩过那种角色扮演游戏里的加点系统？你先把点数分配到力量、敏捷、智力各个属性上，拖来拖去觉得满意了，最后必须按下「确认」按钮，游戏才会真正把这套属性应用到你的角色身上。在那个「确认」按钮按下去之前，你的角色其实还在用初始默认属性跑来跑去。`flag.Parse()` 就是这个「确认」按钮——少了它，你声明的所有 flag 全都只停留在默认值状态，永远不会读到命令行里你传进来的内容。

具体来说，当你用 `flag.String`、`flag.Int`、`flag.Bool` 这些函数声明完所有标志之后，Go 并不会自动去扫描 `os.Args`。它只是把每个 flag 的名称、默认值和描述信息登记在一张内部表格里，静静等待。只有你在 `main` 函数里手动写下 `flag.Parse()` 这一行，Go 才会从 `os.Args[1:]` 开始逐个解析命令行参数，把用户实际传入的值塞进对应的指针地址。这个过程很像微信红包：你把金额填好、备注写完，但必须点「塞钱到红包」，钱才真正装进去，收到的人才能看到实际金额，而不是一个空红包。

因此，`flag.Parse()` 的调用位置有严格要求：必须放在「声明所有 flag」之后，同时放在「第一次读取任何 flag 值」之前。如果你在 `flag.Parse()` 执行前就去解引用那些指针，拿到的只会是你设置的默认值，程序不会报错，但行为完全不符合预期，这类 bug 排查起来相当费时。另外，一个程序里只需要调用一次 `flag.Parse()`，重复调用既没有任何额外效果，也会让代码显得冗余，养成「声明完毕立即在 main 入口处调用一次」的习惯就够用了。

### 解引用和剩余参数

小师弟，有个小坑你要注意一下，用 `flag.String`、`flag.Int`、`flag.Bool` 这几个函数注册标志的时候，它们返回的其实是**指针**，不是值本身。很多新手写完之后直接拿变量去打印，结果输出的是一串内存地址，摸不着头脑。正确做法是在变量前面加一个星号做解引用，比如 `*name`、`*port`、`*verbose`，这样才能拿到用户实际传进来的内容。

如果你觉得每次都要记得加星号太麻烦，可以改用 `flag.StringVar`、`flag.IntVar`、`flag.BoolVar` 这一族函数。它们的区别在于，你自己先声明一个普通变量，然后把它的地址传给函数，解析完成之后直接读那个变量名就行，Go 帮你把解引用这步做掉了，读起来更顺眼，也不容易出错。两种写法功能完全一样，选自己顺手的就好。

说完指针，再说另一件事：`flag.Args()`。`flag.Parse()` 执行之后，命令行里那些**不属于任何标志**的参数，会被统一收集起来，通过 `flag.Args()` 以字符串切片的形式返回。举个你熟悉的例子，`go build -v main.go` 里的 `main.go` 就是位置参数，`-v` 才是标志。你可以把这个过程想象成进门脱鞋：脱下来的鞋是标志，被程序认领走了；脱完鞋之后桌上还放着的钱包、钥匙、手机，就是 `flag.Args()` 返回的那些剩余内容，程序可以按顺序逐个取用。写需要接收文件路径或其他自由参数的小工具时，这个函数会非常有用，记得在 `flag.Parse()` 调用之后再去读它，否则切片还是空的。

### 标志的写法规矩

Go 的 `flag` 包和你以前用过的传统 Unix 工具在命令行解析上有些细微差别，师弟你一定要留心，不然程序跑起来会出现莫名其妙的问题。

首先说说横杠的写法。`flag` 包对单横杠和双横杠都一视同仁，也就是说 `-word foo`、`--word foo`、`-word=foo` 这三种写法全部合法，解析结果完全一样。你用哪种都行，但建议团队内部统一风格，免得代码审查时有人纠结。

然后是 `bool` 类型的标志。和其他类型不同，`bool` 标志可以省略后面的值，直接写 `-fork` 就表示把这个标志置为 `true`，不需要写成 `-fork=true`。当然你想写完整也没问题，`-fork=false` 同样有效。这个设计是为了让开关型参数写起来更简洁。

最重要的一点，师弟你务必记住：`flag` 包一旦遇到它不认识的标志，就会立刻停止解析，把剩余的所有内容统统当作位置参数处理，而不是报错继续往后找。这和 GNU 风格的解析器行为不同，GNU 工具会跳过不认识的部分继续扫描后面的标志。正因如此，你的标志必须写在位置参数的前面，绝对不能把 `-verbose` 这类标志夹在文件名中间，否则从那个文件名开始，后面所有的标志都会被当成普通参数传进去，程序根本不会按你期望的方式运行。养成「标志在前、参数在后」的习惯，可以帮你省去很多排查问题的时间。

## 🏃 跑一下试试

先把程序编译成二进制，再运行，这样才能真实体验命令行标志的效果。

**示例一：把四个标志都传进去**

```shell
go build -o command-line-flags .
./command-line-flags -word=opt -numb=7 -fork -svar=flag
```

```text
word: opt
numb: 7
fork: true
svar: flag
tail: []
```

`-fork` 不带值直接写，布尔标志就会被置为 `true`。四个标志按声明顺序依次打印，`tail` 为空说明没有剩余参数。

**示例二：只传部分标志，后面跟普通参数**

```shell
./command-line-flags -word=opt a1 a2 a3
```

```text
word: opt
numb: 42
fork: false
svar: bar
tail: [a1 a2 a3]
```

注意 `numb`、`fork`、`svar` 都回到了默认值。`a1 a2 a3` 出现在第一个非标志参数的位置，`flag.Parse()` 遇到它们就停止解析，把它们全部塞进 `flag.Args()` 返回的切片，也就是 `tail`。

**示例三：触发自动帮助文档**

```shell
./command-line-flags --help
```

```text
Usage of ./command-line-flags:
  -fork
    	a bool (default false)
  -numb int
    	an int (default 42)
  -svar string
    	a string var (default "bar")
  -word string
    	a string (default "foo")
```

`--help` 是 `flag` 包内置支持的，不需要手动声明。帮助信息会列出全部标志、类型以及默认值，非常省心。学弟如果用 `flag.String` 或 `flag.StringVar` 时填了有意义的 usage 字符串，这里显示出来就更友好了。

## 💡 师兄的碎碎念

**师兄碎碎念**

Go 的 `flag` 包和 POSIX 风格整体上比较兼容：`-name=value` 和 `--name=value` 都能认，双横线和单横线效果一样。但有几个坑要提前踩清楚。

第一，Go flag **不支持** POSIX 的合并短选项。`ls -la` 在 shell 里是 `-l` 和 `-a` 合在一起，Go 会把 `la` 当成一个完整的标志名去查找，找不到就直接报错。所以用 Go flag 写工具，每个选项都得老老实实分开写。

第二，`flag.Parse()` 遇到**第一个非标志参数**就会立刻停下来，后面的所有内容都被丢进 `flag.Args()`。这意味着 `./app a1 -word=opt` 这种写法里，`-word=opt` 不会被解析为标志，而是作为普通参数出现在 tail 里。顺序很重要，标志一定要放在前面。

第三，默认值的类型要和声明函数匹配。`flag.Int` 的默认值传整数，`flag.String` 传字符串，别手滑传字符串 `"42"` 给 `flag.Int`，编译器会告诉你不对。

第四，布尔标志是个特殊公民。想关掉它，不能写 `-fork false`——`flag` 会把 `false` 当成剩余参数，`fork` 依然是 `true`。必须写成 `-fork=false`，等号不能省。想打开直接写 `-fork` 就够了，不用加 `=true`。记住这个不对称性，能省不少调试时间。
