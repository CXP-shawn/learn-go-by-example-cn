# 第 9 关：切片（师兄带你学 Go）

## 🎯 这一关你会学到

切片（Slice）是 Go 里最常用的数据结构，没有之一，这一关你会彻底搞懂它的创建、追加、复制和切割操作。我们还会深入聊聊切片底层的工作原理，让你以后写代码不再踩那些让人怀疑人生的坑。学完之后，你会发现之前学的数组其实只是个

## 🤔 先想一个问题

你有没有用过那种可以无限续杯的奶茶？点单的时候杯子大小是固定的，但有些店会给你一个「超大桶」，喝完了还能往里加，加满了还能换个更大的桶继续加——这就是切片的感觉。

相比之下，上一关学的数组就像普通的一次性纸杯：买的时候就定好容量，340ml 就是 340ml，一滴都多不了。

**切片（Slice）** 是 Go 语言内置的、长度可动态增长的序列类型。它的底层依托于一个数组，但对外暴露的是一个包含「指向底层数组的指针、当前长度（len）、容量（cap）」三个字段的结构体——也就是说，切片本身只是一个轻量级的「视图」，真正的数据住在底层数组里。正是因为有了这个设计，切片才能在保持高性能的同时，做到随时追加、随时裁切，成为 Go 程序员日常撸代码最离不开的工具。

所以问题来了：**既然切片底层还是数组，那我对一个切片做切割（slice[low:high]）之后，修改新切片的元素，原来那个切片的数据会不会也跟着变？** 带着这个问题往下看，答案就在本关代码里。

## 📖 看代码

	s := make([]string, 3)
	fmt.Println("emp:", s)

## 🔍 师兄给你逐行拆

### 1. 用 `make` 创建切片

```go
s := make([]string, 3)
fmt.Println("emp:", s)
```

**这行在干嘛？**

`make` 是 Go 的内建函数，专门用来创建切片、map、channel 这三种引用类型。这里我们用它创建了一个「元素类型是 string、长度是 3」的切片，并赋值给变量 `s`。打印结果是 `emp: [  ]`——三个空字符串，因为 string 的零值就是 `""`。

**为什么这么写？**

你可能会问，为什么不直接写 `var s []string` 就完事了？答案是：`var s []string` 声明出来的是一个 **nil 切片**，它的长度是 0，底层指针是 nil，此时你去做 `s[0] = "a"` 会直接 panic。而 `make([]string, 3)` 帮你做了两件事：**分配底层数组内存** + **把长度设为 3**，所以三个槽位是真实存在的，可以安全读写。

**生活类比**

`make([]string, 3)` 就像你在奶茶店点了一个「三格分装盒」——三个格子已经存在了，只是现在还是空的（零值）。而 `var s []string` 就像你说「我要一个分装盒」，但店员还没给你拿过来，这时候你直接去往格子里放东西，当然会出事。

**容易踩的坑**

`make` 还有第三个可选参数——**容量（cap）**，写法是 `make([]string, 3, 10)`，意思是「长度 3、容量 10」。容量决定了底层数组有多大，在容量范围内 append 不需要重新分配内存。如果你预先知道切片大概会增长到多大，提前设置容量能显著提升性能。很多新手不知道这个参数，等 append 上万次之后才发现程序慢得像蜗牛。

### 2. 索引读写

```go
s[0] = "a"
s[1] = "b"
s[2] = "c"
fmt.Println("set:", s)
fmt.Println("get:", s[2])
```

**这行在干嘛？**

跟数组完全一样——用下标给每个位置赋值，然后读取。`s[2]` 取出来的是 `"c"`，打印 `get: c`。

**为什么这么写？**

切片的索引从 0 开始，这是 C 系语言的传统。Go 没有搞什么「从 1 开始」的奇怪设计，所以长度为 3 的切片，合法下标是 0、1、2，访问 `s[3]` 会在运行时触发 **索引越界（index out of range）** panic。

**生活类比**

想象你家有个三层置物架，第一层是 0 号、第二层是 1 号、第三层是 2 号，往上就没了。你非要去 3 号层放东西，直接摔一跤——这就是越界 panic。

**容易踩的坑**

新手最常犯的错误是在循环里用 `i <= len(s)` 作为终止条件（多了个等号），导致最后一次循环访问 `s[len(s)]`，必然越界。正确写法是 `i < len(s)`。

---

### 3. `len` 函数

```go
fmt.Println("len:", len(s))
```

**这行在干嘛？**

`len` 是内建函数，返回切片当前的「有效长度」，也就是你能合法访问的元素个数。这里打印 `len: 3`。

**切片的三个核心属性**

这里师兄插播一个重要知识点：每个切片在内存里都是一个包含三个字段的结构体：
- **指针（ptr）**：指向底层数组的起始元素
- **长度（len）**：当前切片包含的元素个数
- **容量（cap）**：从当前起始位置到底层数组末尾的元素总数

可以把它想成一个「外卖窗口」：底层数组是整个仓库，`ptr` 是窗口对着仓库哪个货架，`len` 是这次能给你拿几件，`cap` 是这个货架后面还有几件可以随时补充进来。

`len(s)` 只返回长度，想知道容量要用 `cap(s)`。两者在 append 扩容时的区别非常重要，下面马上讲。

---

### 4. `append` 追加元素（重点！）

```go
s = append(s, "d")
s = append(s, "e", "f")
fmt.Println("apd:", s)
```

**这行在干嘛？**

`append` 把一个或多个新元素追加到切片末尾，返回新切片。追加 `"d"` 之后 s 变成 `[a b c d]`，再追加 `"e"` 和 `"f"` 变成 `[a b c d e f]`。

**为什么必须 `s = append(s, ...)` 接收返回值？**

这是 Go 最容易让新手迷糊的设计之一。原因在于 **append 的扩容机制**：

当你追加元素时，Go 首先检查底层数组还有没有剩余容量（cap - len > 0）。如果有，直接把新元素写进去，返回一个 len 加 1 的新切片头（但指向同一底层数组）；如果没有，Go 会**分配一块更大的新内存**，把旧数据全部复制过去，再追加新元素，然后返回一个指向新内存的全新切片。

关键在于：**append 永远返回新切片值**。如果你不用 `s =` 接收，原来的 `s` 还是老样子，新元素石沉大海。这就像你点了外卖续单，但是没有给快递员新地址，快递还是送到老地方，你当然收不到。

**扩容策略**

Go 的 append 扩容大概遵循以下规律（Go 1.18+ 做了调整，但思路一致）：当容量不足时，通常会把容量翻倍（小切片）或按 1.25 倍增长（大切片）。所以如果你反复 append，你会看到 `cap` 以跳跃的方式增长：1→2→4→8→16……每次扩容都有内存重新分配和数据复制的开销。

**容易踩的坑**

```go
// 危险代码，千万别这么写！
append(s, "d")  // 没有接收返回值，这行等于白写
```

Go 编译器甚至不会报错（只是 warning），所以这个坑特别隐蔽。还有一个坑：append 之后如果没有扩容，新切片和原切片共享底层数组，修改新切片的元素会影响原切片——这就是开头问题的答案！

---

### 5. `copy` 复制

```go
c := make([]string, len(s))
copy(c, s)
fmt.Println("cpy:", c)
```

**这行在干嘛？**

先创建一个和 `s` 等长的新切片 `c`，然后用 `copy` 把 `s` 的全部元素复制进去。打印出来 `cpy: [a b c d e f]`，和 `s` 内容完全一样。

**为什么要 copy，不直接 `c := s` 呢？**

因为 `c := s` 只是复制了切片头（那个三字段结构体），`c` 和 `s` 依然指向**同一块底层数组**。你改 `c[0]`，`s[0]` 也跟着变——这叫**浅拷贝**。而 `copy` 是**深拷贝**，它把底层数据真正复制了一份，`c` 和 `s` 从此互不影响。

**生活类比**

`c := s` 就像你把一份文档的「快捷方式」发给同事，你们打开的是同一个文件，一个人改了另一个人也看到改动。`copy` 就像把文档真正复制了一份发过去，各改各的，互不干扰。

**容易踩的坑**

`copy(dst, src)` 实际复制的元素个数是 `min(len(dst), len(src))`。如果你的目标切片 `c` 创建时长度不够，只会复制一部分，剩下的元素悄无声息地丢失，不会报错。所以一定要确保目标切片的长度足够大，或者用 `copy` 的返回值（实际复制数量）来验证。

### 6. 切片操作 `[low:high]`（三种姿势）

```go
l := s[2:5]
fmt.Println("sl1:", l)

l = s[:5]
fmt.Println("sl2:", l)

l = s[2:]
fmt.Println("sl3:", l)
```

**这行在干嘛？**

`s[2:5]` 从 s 中切出一段——包含下标 2、3、4 的元素（不含 5），结果是 `[c d e]`。`s[:5]` 等价于 `s[0:5]`，从头切到第 5 个（不含），结果 `[a b c d e]`。`s[2:]` 等价于 `s[2:len(s)]`，从第 2 个切到末尾，结果 `[c d e f]`。

**三种写法总结**
| 写法 | 等价于 | 含义 |
|------|--------|------|
| `s[low:high]` | 完整写法 | 下标 low 到 high-1 |
| `s[:high]` | `s[0:high]` | 从头到 high-1 |
| `s[low:]` | `s[low:len(s)]` | 从 low 到末尾 |

**底层共享！重要！**

切片操作不复制数据，`l` 和 `s` 共享同一块底层数组。修改 `l[0]`，会直接影响 `s[2]`。这个特性在传递大切片给函数时非常高效（不需要复制整个数据），但也是很多 bug 的根源。如果你不希望共享，记得用 `copy`。

**生活类比**

把底层数组想象成一条长长的游戏地图，切片操作就是给你一个「观察窗口」，只露出地图的一部分。窗口里发生的变化，地图上是真实存在的。

**容易踩的坑**

`s[low:high]` 要求 `0 <= low <= high <= cap(s)`，注意是 cap 不是 len！所以你可以用 `s[0:cap(s)]` 「看到」底层数组里那些超出 len 但在 cap 以内的元素（append 预留的位置）。这个高级技巧用得不好会读到脏数据，初学者先了解即可。

### 7. 字面量初始化

```go
t := []string{"g", "h", "i"}
fmt.Println("dcl:", t)
```

**这行在干嘛？**

用「切片字面量」一步到位：声明类型 + 分配内存 + 赋初始值，`t` 直接就是 `[g h i]`，长度和容量都是 3。

**和 make 的区别**

`make` 适合「我知道大概需要多少个元素，但初始值用零值就行」的场景；字面量适合「我已经知道初始内容」的场景。实际项目里字面量用得非常频繁，比如初始化配置项、测试数据等。

**容易踩的坑**

注意区分切片字面量 `[]string{"a", "b"}` 和数组字面量 `[2]string{"a", "b"}`——前者方括号里什么都没有，后者里面有长度数字。这两个长得很像，类型却完全不同，函数参数传错了会编译报错。

### 8. 多维切片

```go
twoD := make([][]int, 3)
for i := 0; i < 3; i++ {
    innerLen := i + 1
    twoD[i] = make([]int, innerLen)
    for j := 0; j < innerLen; j++ {
        twoD[i][j] = i + j
    }
}
fmt.Println("2d: ", twoD)
```

**这行在干嘛？**

创建一个「切片的切片」（二维切片），外层有 3 个元素，每个元素又是一个 int 切片。关键是内层切片的长度不一样：第 0 行长 1，第 1 行长 2，第 2 行长 3——这是个「阶梯形」结构。最终打印 `2d: [[0] [1 2] [2 3 4]]`。

**和多维数组的区别**

多维数组要求每行等长，`[3][3]int` 就是 3×3 的矩形。多维切片可以每行不等长，更像「锯齿数组（jagged array）」，灵活得多。在表示稀疏数据、不规则表格时非常好用。

**容易踩的坑**

`make([][]int, 3)` 只创建了外层切片，内层的三个切片都是 nil，必须在循环里逐一 `make` 才能使用。新手常忘了初始化内层，直接赋值导致 panic。另外，外层 `make` 第一个参数是 `[][]int`（元素类型是 `[]int`），初学容易写错成 `[]int`。

### 9. 切片与数组的关系（底层原理总结）

到这里我们把底层原理整体串一遍。**切片本质上是对底层数组的一个引用视图**，它自己在栈上只占 24 字节（64 位系统：8 字节指针 + 8 字节 len + 8 字节 cap），真正的数据在堆上的底层数组里。

这带来几个重要推论：

1. **传切片给函数，不会复制元素数据**，只复制那 24 字节的切片头，所以传大切片是高效的。但函数内修改元素会影响原始数据（因为共享底层数组），这既是特性也是陷阱。

2. **append 触发扩容后，会断开和原数组的联系**，返回的新切片指向全新的内存。所以 append 之后的切片，不一定还和你之前切出来的那些切片共享底层数组——这是很多并发 bug 的根源。

3. **nil 切片和空切片不一样**：`var s []string` 是 nil 切片，`s == nil` 为 true；`s := []string{}` 或 `s := make([]string, 0)` 是空切片，len 为 0 但不是 nil。在 JSON 序列化等场景下，nil 切片会被序列化成 `null`，空切片会被序列化成 `[]`，区别很关键。

**一句话总结**：切片 = 底层数组的窗口 + 动态增长能力。理解了这个，Go 里 90% 的切片 bug 你都能看穿。

## 💡 师兄的碎碎念

- 切片和数组长得像，但切片是**引用类型**，`copy` 之后改动原切片不影响副本，这个坑面试贼爱考 🎯
- `append` 超出容量时会**重新分配底层数组**，所以一定要用返回值覆盖原变量，忘了接收是经典 bug
- 切片操作 `s[low:high]` 里 high 是**不包含**的，初学者老是搞错边界，建议默念
