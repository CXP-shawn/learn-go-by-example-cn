# 第 57 关：随机数（师兄带你学 Go）

## 🎯 这一关你会学到

掌握 math/rand 的基本用法：用 Intn 生成整数随机数、用 Float64 生成浮点随机数，理解如何通过 NewSource + New 创建自定义随机源，以及种子（seed）如何决定整条随机序列，做到「可控随机」与「真·每次不同」都拿捏。

## 🤔 先想一个问题

你写了个抽奖程序，本地测试运行十次，发现每次第一个抽到的都是 3 号员工。3 号员工当场怀疑人生，以为你内定。实际上你只是忘了设种子——Go 的 math/rand 默认种子是固定的，每次启动跑出来的序列一模一样，就像一台永远从同一个洗牌顺序开始的麻将机，说是随机，其实早就内定好了。今天师兄带你彻底搞清楚这台「麻将机」的开关在哪里。

## 📖 看代码

```go
// Go 的 `math/rand` 包提供了[伪随机数](http://en.wikipedia.org/wiki/Pseudorandom_number_generator)生成器。

package main

import (
	"fmt"
	"math/rand"
	"time"
)

func main() {

	// 例如，`rand.Intn` 返回一个随机的整数 n，且 `0 <= n < 100`。
	fmt.Print(rand.Intn(100), ",")
	fmt.Print(rand.Intn(100))
	fmt.Println()

	// `rand.Float64` 返回一个64位浮点数 `f`，且 `0.0 <= f < 1.0`。
	fmt.Println(rand.Float64())

	// 这个技巧可以用来生成其他范围的随机浮点数，
	// 例如，`5.0 <= f < 10.0`
	fmt.Print((rand.Float64()*5)+5, ",")
	fmt.Print((rand.Float64() * 5) + 5)
	fmt.Println()

	// 默认情况下，给定的种子是确定的，每次都会产生相同的随机数数字序列。
	// 要产生不同的数字序列，需要给定一个不同的种子。
	// 注意，对于想要加密的随机数，使用此方法并不安全，
	// 应该使用 `crypto/rand`。
	s1 := rand.NewSource(time.Now().UnixNano())
	r1 := rand.New(s1)

	// 调用上面返回的 `rand.Rand`，就像调用 `rand` 包中函数一样。
	fmt.Print(r1.Intn(100), ",")
	fmt.Print(r1.Intn(100))
	fmt.Println()

	// 如果使用相同种子生成的随机数生成器，会生成相同的随机数序列。
	s2 := rand.NewSource(42)
	r2 := rand.New(s2)
	fmt.Print(r2.Intn(100), ",")
	fmt.Print(r2.Intn(100))
	fmt.Println()
	s3 := rand.NewSource(42)
	r3 := rand.New(s3)
	fmt.Print(r3.Intn(100), ",")
	fmt.Print(r3.Intn(100))
}
```

## 🔍 师兄给你逐行拆

**第一段：rand.Intn(100) —— 最常用的整数随机**

先从最简单的说起。`rand.Intn(100)` 会返回一个 int，范围是 `[0, 100)`，也就是 0 到 99，左闭右开，100 本身永远不会出现。这个「左闭右开」是 Go 里到处都是的约定，切片、for range 都是这个逻辑，记住就好。

有个大坑要提前说清楚：`rand.Intn(0)` 直接 panic，`rand.Intn(-1)` 也 panic。原因是这俩参数让「随机范围」变得没有意义——你要在 0 个东西里随机选一个，选什么？Go 的处理方式是直接崩给你看，毫不客气。所以在业务代码里如果 n 是变量，一定要先判断 `n > 0` 再调用，不然上线必炸，血的教训。

代码里打印了两次 `rand.Intn(100)` 用逗号隔开，纯粹是为了演示效果，没什么神奇的。

---

**第二段：rand.Float64() —— 浮点随机，万能的「百分比骰子」**

`rand.Float64()` 返回 `[0.0, 1.0)` 的 float64，也是左闭右开，1.0 本身不会出现。你可以把它想成一个「百分比骰子」，每次扔出来一个 0% 到 99.999...% 之间的小数。

单独用这个不太直接，但它是「映射到任意区间」的基础。

---

**第三段：映射到任意区间 [a, b) —— 公式要背下来**

代码里有一行：`(rand.Float64()*5) + 5`，这是要生成 `[5.0, 10.0)` 范围内的随机浮点数。公式的推导也很直观：

> 目标区间长度 = b - a
> 随机值 = a + rand.Float64() * (b - a)

原始 Float64 给你 `[0, 1)`，乘以区间长度 `(b-a)` 就变成 `[0, b-a)`，再加上起点 `a` 就平移到 `[a, b)`。这个公式适用于任何连续区间，用来做游戏里的「装备掉落概率」、「随机地图高度」都是这个套路，背下来，以后省去很多查资料的时间。

---

**第四段：默认种子是固定的 —— 内定疑云的真相**

这是今天最重要的知识点，也是最多人踩坑的地方。

Go 1.20 之前，`math/rand` 包级别的全局随机器默认种子是 **1**，是的，写死的 1。这意味着每次程序启动，调用 `rand.Intn`、`rand.Float64` 得到的序列完全一致。就像你买了一台彩票摇奖机，但它每次开机摇出来的球顺序都是：3, 17, 42, 5, 8……永远这个顺序，换个说法叫「伪随机」的「伪」就体现在这里——算法本身是确定的，给定相同输入（种子）永远输出相同序列。

本示例代码用的是老式 API，演示的也是这个老行为，所以你在 Go 1.20 之前的版本跑这段代码，第一行两个 `rand.Intn(100)` 每次都是同一对数字。

（Go 1.20 之后，全局随机器自动用随机种子初始化了，包级别的 `rand.Intn` 每次重启都不同，这个坑算是官方填上了。但如果你用的是自定义 Source，行为还是完全由你控制。）

---

**第五段：rand.NewSource + rand.New —— 自己造一台摇奖机**

要解决「每次都一样」的问题，标准做法是用时间戳做种子：

```go
s1 := rand.NewSource(time.Now().UnixNano())
r1 := rand.New(s1)
```

`rand.NewSource(seed)` 创建一个随机源，`seed` 是 int64 类型。`rand.New(src)` 用这个源创建一个独立的 `*rand.Rand` 实例。之后用 `r1.Intn(100)`、`r1.Float64()` 就是在这个独立实例上操作，和全局的 `rand.Intn` 完全隔离，互不影响。

用 `time.Now().UnixNano()` 做种子，是因为纳秒级时间戳每次启动几乎不可能一样，实际上解决了「每次一样」的问题。你把它想成：每次开机都用当前时间重新「洗一次牌」，顺序就不同了。

不过注意：如果你的程序在同一纳秒内创建两个 Source，它们种子相同，序列也相同。高并发场景要留意，不过大多数业务代码不会遇到这个极端情况。

---

**第六段：相同种子 = 完全相同序列 —— 可复现随机是个宝**

代码末尾演示了 `r2` 和 `r3`，两个都用种子 `42` 创建，然后各自调用两次 `Intn(100)`，输出完全一致。这个特性乍一看像 bug，实际上是个很有用的「特性」：

- **测试场景**：你写了一个「随机生成测试数据」的函数，测试跑失败了，你记录下当时的种子，下次重新跑用同一个种子，复现出完全一样的数据，稳稳的排查问题。
- **游戏存档**：很多游戏的「程序化生成地图」就用这个原理，存档只需要存一个种子数字，加载时用同一种子重新生成，地图完全一样，省空间。
- **A/B 测试**：固定种子跑两套算法，保证输入随机序列一致，结果差异只来自算法本身，对比更公平。

种子 42 在编程世界是个梗，来自《银河系漫游指南》里「生命、宇宙及一切的终极答案」，程序员喜欢用它做示例，没有特殊含义，就是个彩蛋。

---

**第七段：安全警告 —— 加密场景别用 math/rand**

`math/rand` 是伪随机，算法是公开的（默认是 Go 内部实现的 LFSR/PCG 类算法），如果攻击者知道种子或者观察到足够多的输出，是可以预测后续序列的。用来生成验证码、Token、密码加密密钥这类场景，`math/rand` 完全不够用。

正确做法是用 `crypto/rand`，它从操作系统的熵源（Linux 的 `/dev/urandom`，Windows 的 `CryptGenRandom`）读取真随机数，不可预测，但速度比 `math/rand` 慢一些。

一句话总结：**游戏掉落概率、测试数据生成、随机排序 → math/rand；密码、Token、验证码 → crypto/rand。搞混了是安全漏洞，不是性能问题。**

---

**第八段：Go 1.20+ 的变化，简单提一嘴**

Go 1.20 之后，包级别的 `rand.Intn` 等函数底层已经自动随机化种子，每次重启结果不同，不需要手动设置种子了。但 `rand.NewSource` + `rand.New` 这套自定义 API 依然有价值：

1. 你需要可复现的随机序列时，自定义 Source 固定种子，行为稳定可控。
2. 高并发场景，每个 goroutine 用独立的 `rand.Rand` 实例，避免全局锁竞争，性能更好。
3. 明确控制随机算法和种子，代码意图更清晰。

所以这套 API 不是过时了，而是「默认行为更合理了，自定义需求依然靠它」。

---

**小结**

| 场景 | 推荐做法 |
|------|----------|
| 快速随机整数 | `rand.Intn(n)`，n 必须 > 0 |
| 随机浮点 [0,1) | `rand.Float64()` |
| 随机浮点 [a,b) | `a + rand.Float64()*(b-a)` |
| 每次启动不同序列 | `rand.NewSource(time.Now().UnixNano())` |
| 可复现随机（测试/调试）| `rand.NewSource(固定数字)` |
| 加密/安全场景 | `crypto/rand`，别碰 math/rand |

记住这张表，`math/rand` 的日常用法基本全覆盖了。
