## 📖 看代码

```go
// 想要写出好的 Go 程序，单元测试是很重要的一部分。
// `testing` 包为提供了编写单元测试所需的工具，写好单元测试后，我们可以通过 `go test` 命令运行测试。

// 为方便演示，例子的代码位于 `main` 包，实际上，单元测试的代码可以位于任何包下。
// 测试代码通常与需要被测试的代码位于同一个包下。

package main

import (
	"fmt"
	"testing"
)

// 我们要测试下面这个简单的函数——返回最小值。
// 一般地，需要被测试的代码应该在类似于 `intutils.go` 的文件下，
// 其对应的测试文件应该被命名为 `intutils_test.go`。
func IntMin(a, b int) int {
	if a < b {
		return a
	}
	return b
}

// 通常编写一个名称以 `Test` 开头的函数来创建测试。
func TestIntMinBasic(t *testing.T) {
	ans := IntMin(2, -2)
	if ans != -2 {
		// `t.Error*` 会报告测试失败的信息，然后继续运行测试。
		// `t.Fatal*` 会报告测试失败的信息，然后立即终止测试。
		t.Errorf("IntMin(2, -2) = %d; want -2", ans)
	}
}

// 单元测试可以重复，所以会经常使用 *表驱动* 风格编写单元测试，
// 表中列出了输入数据，预期输出，使用循环，遍历并执行测试逻辑。
func TestIntMinTableDriven(t *testing.T) {
	var tests = []struct {
		a, b int
		want int
	}{
		{0, 1, 0},
		{1, 0, 0},
		{2, -2, -2},
		{0, -1, -1},
		{-1, 0, -1},
	}

	for _, tt := range tests {
		// t.Run 可以运行一个 "subtests" 子测试，一个子测试对应表中一行数据。
		// 运行 `go test -v` 时，他们会分开显示。
		testname := fmt.Sprintf("%d,%d", tt.a, tt.b)
		t.Run(testname, func(t *testing.T) {
			ans := IntMin(tt.a, tt.b)
			if ans != tt.want {
				t.Errorf("got %d, want %d", ans, tt.want)
			}
		})
	}
}

// 基准测试通常在"_test.go"文件中，并以 `Benchmark` 开头命名。
// `testing` 运行器多次执行每个基准测试函数，并在每次运行时增加 `b.N`，
// 直到它收集到精确的测量值。
func BenchmarkIntMin(b *testing.B) {
	// 通常，基准测试运行一个函数，我们在一个 `b.N` 次的循环内进行基准测试。
	for i := 0; i < b.N; i++ {
		IntMin(1, 2)
	}
}
```

## 🔍 师兄给你逐行拆

好，今天我们来聊聊 Go 语言里 testing 包的基础用法。这是很多初学者容易跳过的部分，觉得「先把功能写完再说测试」，结果功能越堆越多，最后测都不知道从哪下手。所以我强烈建议你从一开始就养成写测试的习惯，而 Go 在这方面其实给你铺好路了。

首先最值得高兴的一点：testing 是 Go 标准库自带的包，你不需要去找什么第三方框架，不用 go get，不用纠结用哪个断言库，打开文件直接 import "testing" 就完事。这和其他语言比起来真的省心不少，不然你光是选测试框架就能纠结半天。

我们先看一个最简单的例子。假设我们有一个 IntMin 函数，它接受两个 int 类型的参数，然后返回其中较小的那个。逻辑非常直白，if 判断一下，谁小返回谁。这种函数你随手就能写出来，但问题是：你怎么确认它真的对？靠肉眼看？靠手动跑一遍 main 函数打印结果？这些方式都不优雅，也不可靠，更没法自动化跑。

这时候 testing 包就登场了。我们来写对应的测试函数 TestIntMinBasic。这个函数接受一个参数，类型是 testing.T 的指针，也就是 *testing.T，我们通常把它命名为 t。在函数体里，我们调用 IntMin，传入 2 和负 2，然后把结果存起来。接下来判断：如果结果不等于负 2，就调用 t.Errorf 来报告一个错误，把期望值和实际值都打印出来，方便你定位问题。

说到这里，有几个硬性规定你必须记住，不然 Go 的测试工具链根本找不到你的测试。

第一，测试文件的命名必须以 _test.go 结尾。比如你的业务代码叫 math.go，那测试文件就叫 math_test.go。这个下划线加 test 加点 go 的后缀是工具链识别测试文件的唯一标志，缺了任何一个字符都不行。而且测试文件要和被测代码放在同一个包里，这样才能直接访问包内的函数。

第二，测试函数的命名必须以大写的 Test 开头，后面跟着你自己起的名字，比如 TestIntMinBasic、TestAdd、TestParseURL 都行。函数签名只能接受一个参数，就是 *testing.T，不能多也不能少，否则 go test 不认识它。

第三，你不需要自己注册任何东西。你只要按规矩写好文件名和函数名，在项目目录下跑 go test 命令，工具链会自动扫描所有符合规范的测试函数并依次执行。这比某些语言里要手动注册测试套件方便太多了，省去了大量样板代码。

接下来我们再说说 t.Errorf 和 t.Fatalf 的区别，这俩很容易搞混。t.Errorf 的意思是：「这里有个错误，记下来，但是继续往下跑。」也就是说，即使这一行报了失败，当前测试函数里后面的代码依然会执行。而 t.Fatalf 则是：「这里有个致命错误，立刻标记本函数为 FAIL，然后直接退出这个测试函数，后面的代码不跑了。」什么时候用哪个？一般来说，如果后续的断言依赖前面某个值是正确的，那就用 Fatalf，免得后面一连串的错误都是因为同一个根本原因。如果各个断言彼此独立，用 Errorf 更合适，能让你一次看到所有失败点。

然后是 t.Log。它的作用是在测试过程中记录一些日志信息，比如你想打印某个中间值方便调试。但是注意，t.Log 记录的内容默认是不显示的，你跑 go test 正常输出里看不到它。只有在运行 go test 时加上 -v 参数，也就是 verbose 模式，这些日志才会被打印出来。所以平时写测试的时候可以适当加一些 t.Log，不会干扰正常的测试输出，调试的时候加个 -v 就能看到详细信息，非常实用。

还有一个点很多人不知道：每个测试函数实际上是运行在独立的 goroutine 里的。这意味着不同测试函数之间天然不共享状态，你不用担心测试 A 改了某个全局变量影响测试 B。当然，如果你自己在测试里用了全局变量，那另说，那是你自己挖的坑。

最后聊一下子测试。如果你想在一个测试函数里跑多个小场景，可以用 t.Run 来创建子测试。t.Run 接受一个名字和一个函数，那个函数同样接受 *testing.T 作为参数。子测试不仅让结构更清晰，还可以配合 t.Parallel 实现并发跑，进一步加快测试速度。不过初学阶段先把基础用熟，子测试并发这块等你写的测试多了再去深入研究。

总结一下：testing 包开箱即用，文件以 _test.go 结尾，函数以 Test 开头，go test 自动发现执行，Errorf 继续跑 Fatalf 立即停，Log 加 -v 才可见，测试间互不干扰。把这几条记牢，你已经能写出像样的 Go 测试了。

好，我们继续。上一段讲了基础的 Test 函数写法，这一段来把测试技巧拉满。

**表驱动测试**

实际项目里，一个函数往往有很多边界情况要验证。如果每个 case 都写一个独立的测试函数，代码会非常啰嗦。Go 社区普遍采用

## 🏃 跑一下试试

在终端先跑 `go test -v`，你会看到 TestIntMinBasic、TestIntMinTableDriven 以及它的每条子测试（如 --- PASS: TestIntMinTableDriven/0,1 等）依次打印 PASS，最后一行显示 ok 和包名。确认绿了之后，再跑 `go test -bench=.`，Go 会自动调整循环次数直到结果稳定，输出类似 `BenchmarkIntMin-8   1000000000   0.24 ns/op`，其中 -8 表示 GOMAXPROCS=8，后面是每次操作耗时。如果只想跑基准不跑单测，加 `-run=^$` 屏蔽单测噪音：`go test -run=^$ -bench=.`。整个流程不超过三条命令，结果一目了然。

## 💡 师兄的碎碎念

💡 测试小贴士
1. 文件名必须以 `_test.go` 结尾，Go 工具链才会识别，普通构建时自动忽略，不会污染生产二进制。
2. 表驱动测试是 Go 社区最主流的姿势：把所有用例塞进结构体切片，一个循环搞定，新增用例只需加一行数据，维护成本极低。
3. `t.Run` 开启子测试后，可以用 `go test -run TestIntMinTableDriven/负数` 单独跑某条 case，调试神器。
4. `go test -cover` 一键输出覆盖率百分比，`-coverprofile=c.out` 再配合 `go tool cover -html=c.out` 可视化哪行没被测到。
5. Benchmark 里的循环次数永远写 `b.N`，不要硬编码 1000 或 100000——Go 运行时会自动放大 N 直到计时结果稳定，你的数字只会帮倒忙。
