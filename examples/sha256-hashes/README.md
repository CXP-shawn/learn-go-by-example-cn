## 📖 看代码

```go
// [_SHA256 散列（hash）_](https://en.wikipedia.org/wiki/SHA-2)
// 经常用于生成二进制文件或者文本块的短标识。
// 例如，TLS/SSL 证书使用 SHA256 来计算一个证书的签名。
// 这是 Go 中如何进行 SHA256 散列计算的例子。

package main

// Go 在多个 `crypto/*` 包中实现了一系列散列函数。
import (
	"crypto/sha256"
	"fmt"
)

func main() {
	s := "sha256 this string"

	// 这里我们从一个新的散列开始。
	h := sha256.New()

	// 写入要处理的字节。如果是一个字符串，
	// 需要使用 `[]byte(s)` 将其强制转换成字节数组。
	h.Write([]byte(s))

	// `Sum` 得到最终的散列值的字符切片。`Sum` 接收一个参数，
	// 可以用来给现有的字符切片追加额外的字节切片：但是一般都不需要这样做。
	bs := h.Sum(nil)

	// SHA256 值经常以 16 进制输出，例如在 git commit 中。
	// 我们这里也使用 `%x` 来将散列结果格式化为 16 进制字符串。
	fmt.Println(s)
	fmt.Printf("%x\n", bs)
}
```

# Go 标准库 sha256 加密摘要函数使用教学

---

## 甲、摘要函数基础概念

在密码学和工程实践里，「摘要函数」（也叫哈希函数）是一类非常基础却极其重要的工具。它的核心能力可以用一句话概括：**接受任意长度的输入，产出固定长度的输出**。不管你塞进去的是一个字节，还是一部几十 GB 的电影，出来的结果长度永远一样。这个特性让摘要函数在各类场景中都极具价值。

除了「定长输出」，摘要函数还有两个关键性质。第一是**单向性**：给你一段摘要值，你在计算上几乎不可能反推出原始数据，这是它被用于安全领域的根本原因。第二是**雪崩效应**：原始输入哪怕只改动一个比特，输出的摘要值就会发生天翻地覆的变化，两段摘要之间看不出任何规律性的关联。正是这三个性质——定长、单向、雪崩——共同构成了摘要函数的基石，也决定了它能在文件校验、数字签名、去重索引等场景中大显身手。

---

## 乙、sha256.New() 返回 hash.Hash 接口

在 Go 标准库里，摘要算法的入口通常是一个 `New()` 函数，比如 `sha256.New()`。这个函数返回的类型是 `hash.Hash` 接口，定义在 `crypto/hash` 和 `hash` 包中。`hash.Hash` 接口长这样：

```go
type Hash interface {
    io.Writer
    Sum(b []byte) []byte
    Reset()
    Size() int
    BlockSize() int
}
```

你会注意到它**内嵌了 `io.Writer`**，也就是说 `hash.Hash` 本身就满足 `io.Writer` 接口，拥有 `Write(p []byte) (n int, err error)` 方法。这个设计不是偶然的，它让摘要对象可以直接和 Go 生态里所有接受 `io.Writer` 的函数无缝协作，比如 `io.Copy`、`bufio` 等。`Size()` 告诉你最终摘要的字节数，`BlockSize()` 告诉你内部处理块的大小，`Reset()` 则允许你复用同一个对象重新开始计算。整个接口设计极为简洁，却覆盖了绝大多数使用场景，这是 Go 标准库一贯的风格。

---

## 丙、流式接口的优势：大文件分块处理

由于 `hash.Hash` 内嵌了 `io.Writer`，它天然支持**流式写入**，这在处理大文件时优势尽显。假设你要计算一个 10 GB 视频文件的摘要，如果先把整个文件读入内存再一次性计算，内存直接爆掉。正确的做法是分块读取、分批写入：

```go
f, _ := os.Open("bigfile.iso")
defer f.Close()
h := sha256.New()
io.Copy(h, f)  // 直接把文件流接到哈希对象上
bs := h.Sum(nil)
fmt.Printf("%x\n", bs)
```

`io.Copy` 会在内部使用一个固定大小的缓冲区（默认 32KB）反复调用 `h.Write`，每一块数据都被「喂」给哈希对象，最终算出的结果和一次性输入完全一致。这得益于摘要函数内部的迭代压缩结构——它维护着一个内部状态，每次 `Write` 都在这个状态上滚动更新，直到你调用 `Sum` 才最终「拍板」输出结果。流式接口让摘要计算在内存占用上几乎与文件大小无关，这是工程上非常优雅的设计。

---

## 丁、[]byte(s) 字符串强转的本质

示例代码里有一行 `h.Write([]byte(s))`，这里做了一次**字符串到字节切片的类型转换**。很多初学者觉得这是一个「强制类型转换」，其实更准确的说法是「类型转换表达式」，它在语义上是创建了一个新的字节切片，内容是字符串底层字节序列的一份副本（编译器在某些情况下会优化掉这次拷贝）。

Go 的字符串底层是一段**不可变的 UTF-8 字节序列**，由一个指针和一个长度组成。`[]byte(s)` 的结果是一个可变的字节切片，拥有指针、长度和容量三个字段。两者的内存布局不同，所以需要显式转换，编译器不会隐式帮你做这件事，这正是 Go 类型安全的体现。对于摘要计算来说，所有数据最终都是字节，不存在「字符」或「编码」的概念，`Write` 拿到的就是原始字节流，你喂进去什么字节，它就算什么字节，不会做任何编码转换，这一点务必牢记。

---

## 戊、h.Sum(nil) 参数意义详解

`Sum` 方法的签名是 `Sum(b []byte) []byte`，它的语义是：**将当前摘要结果追加到 `b` 之后，返回新的字节切片**。当你传入 `nil` 时，等价于追加到一个空切片，最终返回的就是纯粹的摘要字节序列，这是最常见的用法。

但 `Sum` 还有一个重要细节：**调用 `Sum` 不会改变哈希对象的内部状态**。也就是说，你可以在写入一部分数据后调用 `Sum` 拿到「中间摘要」，然后继续写入更多数据，再调用 `Sum` 拿到最终摘要，整个过程互不干扰。这在某些需要「检查点」的协议实现里非常有用。另外，如果你需要把摘要结果拼接到某个已有的字节切片 `prefix` 后面，直接写 `h.Sum(prefix)` 就行，省去了一次额外的 `append` 操作。返回的字节切片长度固定，等于 `h.Size()` 的值，调用方可以放心地依赖这个长度做后续处理。

---

## 己、fmt.Printf("%x") 十六进制输出

`fmt.Printf("%x\n", bs)` 是把字节切片格式化成**十六进制字符串**输出的标准写法。`%x` 动词会将每个字节转换为两个十六进制字符（0–9, a–f），字节之间没有分隔符，整体是一串连续的十六进制数字。由于摘要函数输出固定字节数，对应的十六进制字符串长度也是固定的，例如一个 32 字节的摘要会输出 64 个十六进制字符。

如果你想要大写的十六进制，用 `%X` 即可。如果想要带空格分隔的形式（方便肉眼核对），可以用 `% x`（`%` 和 `x` 之间有空格）。在很多场景下，你需要把摘要存储为字符串而不是字节切片，这时候用 `fmt.Sprintf("%x", bs)` 拿到字符串，或者使用 `encoding/hex` 包的 `hex.EncodeToString(bs)` 函数，两者效果等价，后者在语义上更明确，适合在正式代码里使用。十六进制表示是摘要值最通用的展示和存储格式，Git 的对象 ID、Docker 镜像 ID 都是这种形式。

## 庚、便捷版 sha256.Sum256(data) 一把梭写法

如果你的数据已经全部在内存里，不需要流式处理，Go 标准库通常还提供一个**一次性计算**的便捷函数，比如 `sha256.Sum256(data)`、`sha256.Sum512(data)` 这类形式（具体函数名视算法而定）。这类函数直接接收一个字节切片，返回一个**固定长度的数组**（注意是数组不是切片），例如 `[32]byte`。

```go
data := []byte("xxx this string")
result := sha256.Sum256(data)  // 返回 [32]byte
fmt.Printf("%x\n", result)
```

返回数组而非切片是一个有意为之的设计：数组长度在编译期确定，可以作为 map 的 key，也方便做等值比较（`result1 == result2` 直接可用）。如果你需要切片形式，`result[:]` 一刀切出来即可。便捷函数内部其实就是 `New()` + `Write()` + `Sum(nil)` 的封装，代码量少、意图清晰，数据量小时优先使用，不必每次都手写三行流式调用。

## 辛、老一代摘要算法的安全警告

说到这里，师兄必须提醒你一件重要的事：**不是所有出现在标准库里的摘要算法都适合用于安全场景**。MD5 和 SHA-1 这两个曾经风靡一时的算法，现在在密码学意义上已经**不再安全**。MD5 的碰撞攻击早在 2004 年就被王小云团队攻破，SHA-1 的实际碰撞（SHAttered 攻击）也在 2017 年由 Google 和 CWI 公开演示。所谓「碰撞」，就是找到两段不同的输入，使它们产生相同的摘要值，这直接破坏了摘要函数的核心假设。

Go 标准库保留这些算法，主要是为了兼容老协议和遗留系统，比如某些 TLS 握手流程、老版本的 Git 对象格式。如果你在写新代码，请毫不犹豫地选择 SHA-256、SHA-3 或 BLAKE2 系列这类更现代的算法。安全领域没有「差不多能用就行」，用错算法的代价可能是整个系统的信任体系崩塌。

## 壬、密码存储与实际应用场景

最后要专门说一个极其常见的误区：**绝对不能用普通摘要函数存储用户密码**。有些同学觉得把密码做个 SHA-256 再存库里已经很安全了，其实远远不够。普通摘要函数的设计目标是「快」，现代 GPU 每秒可以暴力尝试数十亿次 SHA-256，配合彩虹表和字典攻击，弱密码几乎瞬间破解。

正确的做法是使用专门为密码存储设计的**慢哈希函数**，如 bcrypt、scrypt 或 Argon2，并配合**随机盐值**（salt）一起使用。盐值确保相同密码在不同用户之间产生不同的哈希结果，防止批量破解。Go 生态里 `golang.org/x/crypto` 包提供了这些算法的实现，请务必用它们处理密码。

相比之下，普通摘要函数大展拳脚的场景是：**文件去重**（内容寻址存储，相同内容只存一份）、**版本控制对象寻址**（Git 用对象的 SHA 值作为唯一 ID，内容不变则 ID 不变）、**令牌签名**（HMAC 场景，配合密钥对消息做完整性验证）、**数据校验**（下载文件后核对摘要，确认未被篡改或损坏）。在这些场景里，摘要函数是你最顺手的工具，用对地方，才能发挥最大价值。

## 🏃 跑一下试试

执行 `go run sha256-hashes.go` 后，终端打印两行。第一行原样输出字符串 `sha256 this string`。第二行是其 SHA256 摘要的十六进制表示，以 `1af1dfa857bf1d88` 开头，完整共 64 个十六进制字符。

为什么是 64 个字符？SHA256 产生 256 位 = 32 字节的固定长度摘要，每个字节用 2 个十六进制字符表示（高 4 位 + 低 4 位），因此 32 × 2 = 64 字符。`%x` 动词会自动完成这个转换，无需手动遍历。

SHA256 的核心价值在于：**确定性**（同一输入永远得到同一输出）、**雪崩效应**（输入改动一个字节，输出完全不同）、**单向性**（无法从摘要反推原文）。这使它广泛用于文件完整性校验、数字签名以及 TLS 证书签发。

## 💡 师兄的碎碎念

**大文件哈希不要一次性读入内存**，直接用 `io.Copy(h, file)` 流式喂入，hash 对象本身实现了 `io.Writer`，既省内存又简洁。

**SHA256 不等于唯一 ID**。在海量对象（如数十亿条记录）的系统里，理论碰撞概率虽极低，但业务上仍需结合 UUID 或加盐设计来规避风险，不要裸用摘要做主键。
