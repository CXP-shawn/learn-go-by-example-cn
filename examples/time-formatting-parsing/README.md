# 第 56 关：时间的格式化和解析（师兄带你学 Go）

## 🎯 这一关你会学到

掌握 Go 的时间格式化与解析，核心是理解那个「神奇的参考时间」——2006-01-02 15:04:05，能熟练使用 time.Format 输出时间字符串，用 time.Parse 把字符串还原成 time.Time，并学会处理解析失败时的 error。

## 🤔 先想一个问题

你有没有第一次看到 Go 的时间格式化，整个人都懵了？别人家的语言用 yyyy-MM-dd HH:mm:ss，人畜无害，一眼就懂。结果 Go 跑来跟你说「你得用 2006 年 1 月 2 日下午 3 点 4 分 5 秒来做模板」——这啥？这是哪个 Go 核心开发者的生日？还是程序员失恋的纪念日？这一关我们就来彻底搞清楚这个「奇葩」设计背后的逻辑，以及怎么不踩坑。

## 📖 看代码

```go
// Go 支持通过基于描述模板的时间格式化与解析。

package main

import (
	"fmt"
	"time"
)

func main() {
	p := fmt.Println

	// 这是一个遵循 RFC3339，
	// 并使用对应的 `布局`（layout）常量进行格式化的基本例子。
	t := time.Now()
	p(t.Format(time.RFC3339))

	// 时间解析使用与 `Format` 相同的布局值。
	t1, e := time.Parse(
		time.RFC3339,
		"2012-11-01T22:08:41+00:00")
	p(t1)

	// `Format` 和 `Parse` 使用基于例子的布局来决定日期格式，
	// 一般你只要使用 `time` 包中提供的布局常量就行了，但是你也可以实现自定义布局。
	// 布局时间必须使用 `Mon Jan 2 15:04:05 MST 2006` 的格式，
	// 来指定 格式化/解析给定时间/字符串 的布局。
	// 时间一定要遵循：2006 为年，15 为小时，Monday 代表星期几等规则。
	p(t.Format("3:04PM"))
	p(t.Format("Mon Jan _2 15:04:05 2006"))
	p(t.Format("2006-01-02T15:04:05.999999-07:00"))
	form := "3 04 PM"
	t2, e := time.Parse(form, "8 41 PM")
	p(t2)

	// 对于纯数字表示的时间（时间戳），
	// 您还可以将标准字符串格式与提取时间值的一部分一起使用。
	fmt.Printf("%d-%02d-%02dT%02d:%02d:%02d-00:00\n",
		t.Year(), t.Month(), t.Day(),
		t.Hour(), t.Minute(), t.Second())

	// 当输入的时间格式不正确时，`Parse` 会返回一个解析错误。
	ansic := "Mon Jan _2 15:04:05 2006"
	_, e = time.Parse(ansic, "8:41PM")
	p(e)
}
```

## 🔍 师兄给你逐行拆

先从整体说一下这一关在干啥。时间的格式化（Format）就是把一个 time.Time 对象变成一个人类可读的字符串，比如「2024-06-15T10:30:00+08:00」；时间的解析（Parse）是反过来，把一个字符串变回 time.Time 对象。这两件事在实际开发中超级常见——打日志要格式化时间，接收前端或第三方 API 传来的时间字符串要解析，往数据库存时间要格式化。可以说，不搞懂这俩函数，写业务代码会一直被时间问题折磨。

**第一段：最简单的用法——RFC3339 标准格式**

代码第一步是 `t := time.Now()` 拿到当前时间，然后 `t.Format(time.RFC3339)`。RFC3339 是啥？就是「2006-01-02T15:04:05Z07:00」这种格式，是互联网上最通用的时间格式之一，JSON API、OAuth token、K8s 配置文件里到处都是这个格式。`time.RFC3339` 是 time 包里内置的常量字符串，你不用手写，直接用就好。如果你的项目要对外暴露 API，强烈建议统一用 RFC3339，时区信息都带上，省去无数跨时区的坑。

**第二段：time.Parse——两个返回值一个都不能忽视**

```go
t1, e := time.Parse(time.RFC3339, "2012-11-01T22:08:41+00:00")
```

time.Parse 返回两个值：解析成功的 time.Time 和一个 error。这里的 e 在代码里后面被复用了，严格来说这段示例代码没有逐次检查 error，是为了演示简洁，真实项目里千万要检查。解析的第一个参数是「布局」，第二个是「要解析的字符串」，两者格式必须匹配，否则就会报错。`+00:00` 代表 UTC 时区，解析出来的 t1 会是一个带时区信息的时间对象。你打印出来会看到「2012-11-01 22:08:41 +0000 UTC」，Go 会自动帮你标注时区。

**第三段：Go 最反直觉的设计——用真实时间当模板，而不是符号**

好，这是本关最核心、最容易让新人炸掉的地方，务必耐心看完。

其他语言格式化时间用的是「占位符号」：Java/Python 用 yyyy 代表年，MM 代表月，dd 代表日，HH 代表小时，mm 代表分钟。逻辑很直白，就是记住一套符号体系。

Go 不是这么干的。Go 的哲学是：**用一个真实存在的参考时间点当模板**，你怎么写这个参考时间，Format 就怎么输出你的时间。这个参考时间是：

**2006 年 1 月 2 日，周一，下午 3:04:05，时区 MST（山地标准时间，UTC-7）**

写成一行就是：`Mon Jan 2 15:04:05 MST 2006`

为什么是这个时间点？这不是某人的生日，也不是失恋纪念日——这是一个精心设计的「记忆法」。你看：
- 年：2006，即 06
- 月：01（January 是第 1 个月）
- 日：02
- 小时：15（24 小时制），或者 03（12 小时制）
- 分钟：04
- 秒：05
- 时区：-07:00

把它排成一行：`01/02 03:04:05PM '06 -0700`，也就是 1、2、3、4、5、6、7！这是按顺序排列的数字！Go 的设计者 Rob Pike 他们用这个参考时间，让你只要记住「1月2日3点4分5秒2006年，时区偏移-7」就能搞定所有格式。理解了这个，你就不会再觉得奇葩，反而会觉得：卧槽，这也太有设计感了。

**第四段：自定义布局，逐个拆解**

`t.Format("3:04PM")`：这里 3 代表 12 小时制的小时（对应参考时间的 3 点），04 代表分钟，PM 代表上午/下午。输出类似「3:04PM」。注意：这里用的是 3，不是 03，03 是带前导零的版本，3 是不带的。

`t.Format("Mon Jan _2 15:04:05 2006")`：这里的 `_2` 是一个细节——下划线加 2，代表「用空格补齐到两位的日期」，比如 1 号会输出「 1」（前面有个空格），10 号就正常输出「10」。这是为了让日期列对齐，在打印日志、报表的时候很有用。

`t.Format("2006-01-02T15:04:05.999999-07:00")`：这是带微秒和时区偏移的 ISO8601 格式，`.999999` 里面 9 的个数代表精度，Go 会根据实际的纳秒值自动裁剪尾部的零。如果你用 `.000000`（全是 0），则会强制补零到六位。这个细节在需要高精度时间戳的场景（比如金融交易日志）下非常有用。

**第五段：「3 04 PM」解析「8 41 PM」——模板对齐的本质**

```go
form := "3 04 PM"
t2, e := time.Parse(form, "8 41 PM")
```

这里用自定义的布局「3 04 PM」去解析「8 41 PM」。3 对应小时 8，04 对应分钟 41，PM 对应下午。能对上！解析成功，t2 会是一个「只有时间没有日期」的时间对象，年月日会被设置成 Go 的零值基准（0000-01-01），在只需要时分的场景（比如营业时间、闹钟）可以这么用，然后把日期部分替换成你想要的。这里暗藏一个坑：如果你把布局写成「03 04 PM」，03 是带前导零的小时，但「8 41 PM」里的 8 没有前导零，就可能解析失败或者解析出奇怪的结果。所以布局和输入字符串的格式必须严格对应，不是大概对上就行。

**第六段：老派做法——用 fmt.Printf 手动拼时间字符串**

```go
fmt.Printf("%d-%02d-%02dT%02d:%02d:%02d-00:00\n",
    t.Year(), t.Month(), t.Day(),
    t.Hour(), t.Minute(), t.Second())
```

这是另一种格式化方式：直接调 time.Time 的各个分量方法，然后用 fmt.Printf 拼成字符串。%02d 是「不足两位补零」的格式化动词。这种方式在 C 语言时代非常常见，代码更啰嗦，但逻辑非常透明，每一段是什么一清二楚。实际项目里，如果你对 Go 的布局语法还不熟，这种方式是个安全牌，出了问题容易 debug。不过一旦熟了，还是推荐用 Format，更优雅，也不容易写错 %02d 和 %d。

注意这里硬编码了 `-00:00` 时区偏移，实际项目中不建议这么做，因为你的服务器可能在不同时区跑，要么用 `t.Format` 加时区占位符，要么用 `t.UTC()` 先转成 UTC 再格式化。

**第七段：Parse 解析失败——error 的真实面目**

```go
ansic := "Mon Jan _2 15:04:05 2006"
_, e = time.Parse(ansic, "8:41PM")
p(e)
```

布局是 ANSIC 格式（类似「Mon Jan  2 15:04:05 2006」），但输入字符串是「8:41PM」，格式完全对不上。Parse 就会返回一个非 nil 的 error，打印出来会告诉你哪里解析失败了。这在实际场景中非常重要——前端传来的时间字符串、第三方 API 返回的数据，格式五花八门，如果你不检查 error，解析失败时拿到的是 time.Time 的零值（0001-01-01 00:00:00 UTC），然后这个零值被存进数据库，之后你会在数据库里看到一堆「0001 年 1 月 1 日」，调查 bug 调到崩溃。所以：Parse 之后必须判断 error。

**总结与实战建议**

1. 对外 API 统一用 RFC3339，不要自己发明格式。
2. 记住参考时间「2006-01-02 15:04:05 -07:00」，年月日时分秒时区对应 6、1、2、3、4、5、7，顺序记忆。
3. 月份占位是 01（两位带零）或 1（不带零），不要混用；日期是 02 或 _2（空格补位）或 2；小时是 15（24 小时制）或 03/3（12 小时制）。
4. Parse 之后一定检查 error，别让 0001 年悄悄溜进数据库。
5. 如果格式实在记不住，就用 fmt.Printf 拼，总比格式写错强。

## 💡 师兄的碎碎念

① Go 的布局魔法数字是 2006-01-02 15:04:05，这是固定的「参考时间」，你要改成 2020-01-02 就会解析出奇怪结果甚至静默出错。背不住的话记住一句话：「2006年1月2日下午3点4分5秒」，按顺序数 1、2、3、4、5、6 就对了。

② _2 和 02 都是补齐到两位，但 _2 是空格补齐（" 1"），02 是零补齐（"01"），日志里看数字对不齐先查查用的哪个。

③ 大坑预警：time.Parse 如果布局里没有时区信息，解析结果默认是 UTC，不是本地时间。中国服务器上吃过亏的人举手——少 8 小时的 bug 找半天，结果就是这个。

④ 对外 API、日志存储推荐统一用 RFC3339（即 ISO 8601），可读性好、跨语言兼容，不用纠结格式细节直接 time.RFC3339 拿去用。

⑤ fmt.Printf 手动拼时间字符串在性能极度苛刻的热路径上有一点点优势，但代码丑、容易写错，普通业务场景老老实实用 Format 就行，别过早优化。

## 🎓 知识点清单

- time.Format() 用固定参考时间 2006-01-02 15:04:05 作为布局模板，而不是随便写个日期
- time.Parse() 和 Format() 共用同一套布局，解析失败会返回 error，不会 panic
- 预置布局常量如 time.RFC3339、time.ANSIC 等可直接拿来用，省得自己写
- _2 表示空格补齐到两位宽度，02 表示零补齐两位，两者效果肉眼看起来不同
- 只给时间不给日期时，Parse 解析出的年月日默认填充为 0001-01-01，属于 Go 的零值行为
- time.Parse 不带时区信息时默认按 UTC 处理，国内用要注意差 8 小时的坑
- fmt.Printf 手动拼装时间字符串可以灵活控制每个字段，但可读性不如 Format
- 解析错误时 error 信息会明确告诉你在哪卡住了，方便 debug
