# 第 69 关：命令行参数（师兄带你学 Go）

## 🎯 这一关你会学到

- `os.Args` 就是一个字符串切片，程序跑起来时收到的所有命令行参数都装在里面。
- 坑在于下标 0 永远是程序自身的路径，不是你传的第一个参数。
- 用切片语法 `os.Args[1:]` 可以直接甩掉第一项，只留你真正想要的参数。
- 硬编码下标（比如直接写 `os.Args[3]`）很危险，参数少了就直接 panic 崩掉。
- 如果参数逻辑变复杂，下一关的 `flag` 包能让你优雅地处理带名字的选项。

## 🤔 先想一个问题

你有没有好奇过，每次敲 `go run main.go` 或者 `git commit -m "fix bug"` 时，这些程序怎么知道你想让它干什么？秘密就藏在命令行参数里——程序启动的瞬间，终端把你输入的那串文字原样打包塞给它。这一关就来搞清楚 Go 是怎么拿到这些参数的，以及新手最容易踩的下标偏移坑。

## 📖 看代码

```go
// [_命令行参数_](http://en.wikipedia.org/wiki/Command-line_interface#Arguments)
// 是指定程序运行参数的一个常见方式。例如，`go run hello.go`，
// 程序 `go` 使用了 `run` 和 `hello.go` 两个参数。

package main

import (
	"fmt"
	"os"
)

func main() {

	// `os.Args` 提供原始命令行参数访问功能。
	// 注意，切片中的第一个参数是该程序的路径，
	// 而 `os.Args[1:]`保存了程序全部的参数。
	argsWithProg := os.Args
	argsWithoutProg := os.Args[1:]

	// 你可以使用标准的下标方式取得单个参数的值。
	arg := os.Args[3]

	fmt.Println(argsWithProg)
	fmt.Println(argsWithoutProg)
	fmt.Println(arg)
}
```

## 🔍 师兄给你逐行拆

### 为什么可执行程序必须写 `package main`

师弟，你有没有玩过手机游戏？每次打开游戏，第一个出现的是主界面，对吧——上面有「开始游戏」按钮，有角色选择，有进入剧情的入口。操作系统就是靠这个主界面找到游戏的「起点」的，没有主界面，操作系统根本不知道该从哪儿开始运行。`package main` 在 Go 里扮演的就是这个角色。Go 编译器在编译代码的时候，要从成百上千个包里找出「这个程序该从哪儿启动」，它的判断依据就是：哪个包声明了 `package main`，哪个包里就有程序的入口。不写这个，编译器只会把你的代码当成一个普通的工具库，就像游戏里的道具包裹一样，单独拿出来没办法运行，只能被别人调用。所以规则很清楚：只要你想让代码直接跑起来，第一行就必须老老实实写上 `package main`，这是和编译器之间的约定，不能省略。

### os 和 fmt：你的程序和外界打交道的两扇门

师弟，你可以把一个 Go 程序想象成一个刚入职的新员工。`os` 包就是这位新员工和「公司系统」打交道的渠道——他需要知道自己的工号（环境变量）、今天领到了什么任务清单（命令行参数，比如 `os.Args`），以及下班打卡时汇报今天干得好不好（退出码，`os.Exit(0)` 表示顺利完成，非零表示出了岔子）。简单说，凡是和操作系统层面有关的事情，都归 `os` 管。而 `fmt` 包则像是这位员工手里的「广播喇叭」，专门负责把话说出来、格式化打印到屏幕上。`fmt.Println` 是你最常用的那个喇叭口，一句话加个换行就搞定。两个包各司其职：`os` 负责「和系统握手」，`fmt` 负责「对人开口说话」。记住这个分工，以后看到它们俩出现在 import 里，脑子里马上就能对号入座，不会再犯晕了。

### os.Args 是啥，为啥第 0 位不是我的参数？

小师弟，`os.Args` 就是一个 `[]string`，也就是字符串切片，专门用来装命令行里你敲的那些东西。但这里有个让新手翻车的坑——`os.Args[0]` **不是你传的参数**，而是这个可执行文件本身的路径，比如 `./myapp` 或者 `/usr/bin/myapp`。

你把它想成去餐厅点菜：`os.Args` 是整张订单，`os.Args[0]` 是印在最上面的**店名**，不是菜——你真正点的菜，从第 1 道开始算，也就是 `os.Args[1]`、`os.Args[2]` 往后排。这是 Unix/Windows 的老传统，操作系统启动程序时，习惯把程序自己的名字塞到第 0 位，Go 老老实实照搬了。

所以你跑 `go run main.go hello world`，`os.Args[1]` 才是 `"hello"`。别一上来就读 `os.Args[0]` 当参数用，那只会读出你自己的程序路径，坑得你一脸懵。记住：**从 1 开始取，0 留给程序自己**。

### 切片去头，只留干货

Go 的切片语法遵循**左闭右开**规则：`a[start:end]` 取的是从 `start` 开始、到 `end` 之前（不含 `end`）的那一段。打个比方，`a[1:3]` 拿到的是下标 1 和 2 的元素，下标 3 的那个不算你的。这个规则初看有点别扭，但用熟了你会发现它让边界计算特别顺手。

当你省略 `end`，写成 `a[1:]`，Go 就会自动帮你取到切片末尾，一个不落。这意味着**从下标 1 开始一路取到底**，正好跳过了第一个元素。学弟你记住这个姿势，以后能省不少力气。

最典型的使用场景就是处理命令行参数。`os.Args` 是 Go 帮你准备好的字符串切片，里面第一项永远是程序自身的可执行路径，后面才是用户真正传进来的参数。所以只要写 `os.Args[1:]`，就能干净利落地甩掉那个多余的路径，只留下你真正关心的参数列表，不用手动算长度，也不用额外声明新变量。

最后有一点要特别叮嘱：**切片不是复制**。`os.Args[1:]` 并不会把数据重新拷贝一份，它和 `os.Args` 共享同一块底层数组。改动新切片里的元素，原切片也会跟着变。大多数情况下这没问题，但如果你打算对参数做修改又不想影响原始数据，记得用 `copy` 主动拷贝一份出来，这才是稳妥的做法。

### 硬编码下标：别学游戏外挂一样越界崩溃

师弟，你写 `os.Args[3]` 的时候，心里有没有想过「万一用户没传第三个参数呢」？没想过对吧，我就知道。这就像你用游戏外挂直接读固定内存地址，地址没数据，直接崩给你看——Go 也一样，运行时直接甩你一脸 `panic: runtime error: index out of range`，程序当场去世，一点面子不给。

`os.Args` 是个普通切片，`os.Args[0]` 是程序名，后面才是你传进来的参数。你硬写 `os.Args[3]`，要求至少得有 4 个元素，但用户少传一个，你的程序就爆。所以**每次访问前，先判断 `len(os.Args)`**，这是基本素养：

```go
if len(os.Args) < 4 {
    fmt.Println("参数不够，请检查输入")
    return
}
```

养成这个习惯，比事后找 panic 日志省心一百倍，记住了没？

### 小预告：下一关见 flag 神器

用 `os.Args` 的切片语法来解析命令行参数，感觉就像用筷子拆手机——能干，但真的很费劲。当你只需要读一个参数的时候，手撸切片还算清爽；可一旦参数种类多起来，还要处理默认值、打印帮助提示，代码就会迅速膨胀成一坨充满 `if-else` 的噩梦。你得自己判断参数顺序、自己校验类型、自己拼帮助文本，写到后面连自己都不想维护了。

别慌，Go 标准库早就替你想好了，这个救星叫做 `flag` 包。它专门用来解析命令行参数，一行声明就能同时搞定参数名、参数类型、默认值和帮助文案，简洁到让人感动。比如 `flag.Int("port", 8080, "服务端口号")` 这一句，该有的全有了，再也不用自己写一堆体力活代码。

不过学长要说一句掏心窝的话：先别急着跳到 `flag`，把 `os.Args` 的切片思路真正吃透才是正道。理解了底层的参数是怎么传进来的、怎么被切割和遍历的，你再去用 `flag`，才会真正明白它替你省了多少麻烦，用起来也会更有底气。基础打扎实，工具用得才顺手。

下一关我们就正式上 `flag` 包的具体用法，绝对值得期待。

## 🏃 跑一下试试

**方法一：先编译再运行（推荐）**

```shell
go build command-line-arguments.go
./command-line-arguments a b c d
```

预期输出：

```text
[./command-line-arguments a b c d]
[a b c d]
c
```

**方法二：直接 `go run`**

```shell
go run command-line-arguments.go a b c d
```

预期输出：

```text
[/tmp/go-build123456789/b001/exe/command-line-arguments a b c d]
[a b c d]
c
```

注意 `go run` 时程序路径会是系统临时目录下的随机路径，每次都不同，但参数部分完全一致。

**方法三：故意少给参数，观察 panic**

```shell
./command-line-arguments a b
```

```text
panic: runtime error: index out of range [3] with length 3
```

程序访问 `os.Args[3]` 时，如果实际传入的参数（含程序名）少于 4 个，Go 运行时会直接 panic 并打印错误信息。这是初学者最容易翻车的地方，生产代码一定要先检查长度。

## 💡 师兄的碎碎念

师兄碎碎念来了。

首先聊聊 Windows 用户的小困扰：在 Windows 下 `go build` 生成的可执行文件默认带 `.exe` 后缀，所以你要敲的是 `.\command-line-arguments.exe a b c d`，别照着教程直接复制 `./command-line-arguments` 然后傻眼找不到文件。这个细节坑过不少刚入门的同学。

然后说说 `go run` 的路径问题。很多人第一次看到 `os.Args[0]` 输出一长串 `/tmp/go-build随机数字/...` 会吓一跳，以为哪里出错了。其实 `go run` 的本质是先把代码编译到临时目录，再运行二进制，所以路径看起来很奇怪很长，这是正常现象，不用慌。如果你的程序逻辑依赖 `Args[0]` 的路径来定位配置文件，记得用 `go build` 出来的二进制测，别用 `go run`，否则路径完全不可预测。

再说参数顺序的问题。`os.Args` 里参数的顺序就是你在命令行敲的顺序，没有任何魔法重排，所以代码里写死下标 `os.Args[3]` 取的永远是第三个用户参数，不是某个特定名字的参数。这种写法极其脆弱，顺序一变全乱。

最后一个大坑：参数里有空格怎么办？Shell 按空格分词，`./prog hello world` 会被解析成两个参数。如果你想传一个带空格的字符串，必须加引号：`./prog "hello world"`，否则空格会把它劈成两截，`os.Args` 里拿到的就是两个独立参数，调试半天找不到原因，真的很崩溃。
