# 第 72 关：环境变量（师兄带你学 Go）

## 🎯 这一关你会学到

- **环境变量的作用**：环境变量是操作系统提供的键值对，进程启动时从父进程继承，常用于传递配置（如数据库地址、端口号、鉴权密钥）而无需硬编码进代码。
- **三个核心 API**：`os.Setenv(key, value)` 在当前进程内设置变量；`os.Getenv(key)` 读取变量值；`os.Environ()` 返回当前进程所有环境变量的字符串切片，每条形如 `KEY=VALUE`。
- **Getenv 空串陷阱**：变量未设置和变量被设为空串，`os.Getenv` 的返回值都是 `""`，无法区分——如果业务上需要区分，必须换用 `os.LookupEnv`。
- **LookupEnv 更严谨**：`os.LookupEnv` 返回 `(value string, ok bool)`，`ok` 为 `false` 才代表变量真的不存在，这在「空串也是合法值」的场景下至关重要。
- **strings.SplitN 按第一个等号切**：环境变量的 value 本身可能包含 `=`（如 Base64 编码的密钥），用 `strings.SplitN(e, "=", 2)` 限定最多切成两段，可以正确保留 value 中的等号。

## 🤔 先想一个问题

你有没有想过，每次打开终端，`echo $PATH` 就能看到一长串路径，`echo $HOME` 就知道用户主目录在哪——这些信息是谁告诉你的程序的？答案是：操作系统在启动进程时，会把一组键值对 塞进进程的内存空间，这就是环境变量。子进程默认继承父进程的所有环境变量，所以你在 shell 里设的 `PATH`，启动的每一个子进程都能读到。Go 的 `os` 包把这套机制封装得非常干净，几行代码就能读、写、遍历所有变量，这一关我们来把它摸透。

## 📖 看代码

```go
// [_环境变量_](http://zh.wikipedia.org/wiki/%E7%8E%AF%E5%A2%83%E5%8F%98%E9%87%8F)
// 是一种[向 Unix 程序传递配置信息](http://www.12factor.net/config)的常见方式。
// 让我们来看看如何设置、获取以及列出环境变量。

package main

import (
	"fmt"
	"os"
	"strings"
)

func main() {

	// 使用 `os.Setenv` 来设置一个键值对。
	// 使用 `os.Getenv`获取一个键对应的值。
	// 如果键不存在，将会返回一个空字符串。
	os.Setenv("FOO", "1")
	fmt.Println("FOO:", os.Getenv("FOO"))
	fmt.Println("BAR:", os.Getenv("BAR"))

	// 使用 `os.Environ` 来列出所有环境变量键值对。
	// 这个函数会返回一个 `KEY=value` 形式的字符串切片。
	// 你可以使用 `strings.SplitN` 来得到键和值。这里我们打印所有的键。
	fmt.Println()
	for _, e := range os.Environ() {
		pair := strings.SplitN(e, "=", 2)
		fmt.Println(pair[0])
	}
}
```

## 🔍 师兄给你逐行拆

### 什么是环境变量？

小师弟，在聊 Go 怎么读写环境变量之前，咱们先把概念搞清楚。

**环境变量（Environment Variable）本质上是操作系统维护的一组全局键值对。** 你可以把它想象成写字楼一楼大堂里那块电子信息屏——整栋楼的所有公司员工进门都能看到上面的内容，但每家公司只关心和自己相关的那条通知，其余的直接忽略。操作系统就是那栋写字楼，每个运行中的进程就是里面的一家公司，而信息屏上滚动的每一行「键=值」就是一条环境变量。

在 Linux 和 macOS 上，你打开终端输入 `env` 或 `printenv`，就能看到当前 Shell 进程持有的所有环境变量。最常见的几个是：`PATH`（告诉系统去哪些目录里找可执行文件）、`HOME`（当前用户的主目录路径）、`USER`（当前登录的用户名）。这些变量在系统启动时就已经准备好了，之后每次你启动一个新进程，它会**继承父进程的整套环境**，就像公司新开一个部门，默认沿用总部的规章制度。

正因为环境变量天然具备「进程隔离、外部注入、无需改代码」的特点，业界著名的 **12-factor App 原则**明确推荐：把数据库地址、API Key、第三方服务 URL 等配置统统放进环境变量，而不是硬编码在源码里。这样一来，同一份代码部署到开发环境（dev）、预发环境（staging）还是生产环境（prod），只需要在各自的宿主机或容器里设置不同的环境变量值，程序本身完全不用动。

好，概念清楚了，下面咱们就来看 Go 标准库里的 `os` 包是怎么帮你读写这块「信息屏」的。

### os.Setenv：给当前进程临时塞个变量

Go 标准库里的 `os.Setenv` 函数签名非常简洁，它只接受两个字符串参数——`key` 和 `value`，调用之后便会在当前运行中的 Go 进程内部写入一条新的环境变量，或者覆盖掉同名的旧值。这个写入操作的作用范围严格限定在当前进程以及由它 fork 出来的子进程之中，一旦程序退出，这条记录就随之消失，绝对不会反向写回到启动它的父 shell 里去。用一个生活化的比喻来理解这件事会更直观：就像你出门前临时戴了一顶帽子，在外面的所有人都能看到你戴着帽子，但你一回到家把帽子摘掉放进柜子，那顶帽子就好像从来没存在过；你的父进程，也就是那个启动程序的 shell，始终处于家里，它从头到尾都看不到你出门时戴的这顶帽子，更不知道你换过什么颜色的款式。正因为这种「用完即走、不污染外部环境」的特性，`os.Setenv` 在自动化测试中被频繁使用，测试代码可以在函数开头临时伪造一个环境变量，模拟特定的运行条件，测试结束后再调用 `os.Unsetenv` 将其清理掉，整个过程对其他测试用例或外部 shell 没有任何副作用。除了测试场景，它同样适合在程序初始化阶段动态覆盖默认配置，例如根据某个配置文件的读取结果，在真正加载业务逻辑之前把关键的环境变量补充完整，从而让后续依赖环境变量的第三方库能够拿到正确的值。需要特别注意的是，`os.Setenv` 并非线程安全的原子操作，如果多个 goroutine 并发地读写同一个环境变量，应当在业务层面做好同步控制，避免出现竞态条件导致的不可预期行为。

### Getenv：拿不到就是空串

Go 标准库的 `os` 包提供了 `Getenv` 函数，专门用于读取当前进程的环境变量，调用方式非常简洁，只需传入一个字符串类型的 key，函数就会返回对应的字符串值。如果这个 key 在系统环境中从未被设置过，`Getenv` 既不会返回错误，也不会触发 panic，而是悄悄地返回一个空字符串，调用方完全感知不到这个 key 是否真的存在于环境中。这种设计在大多数场景下很方便，但在需要区分"变量未设置"和"变量设置为空字符串"这两种含义时就会显得力不从心。

如果你的业务需要精确区分这两种情况，标准库还准备了另一个兄弟函数 `LookupEnv`，它会同时返回 value 和一个布尔值：布尔值为 `false` 时才代表该 key 确实不存在于环境中，`true` 则表示 key 存在且对应的 value 就是你拿到的那个字符串（即便它恰好是空的）。

举个真实世界里常见的翻车案例：你在代码里用 `Getenv` 读取数据库连接字符串这类配置，一旦部署时漏配了对应的环境变量，`Getenv` 不会告诉你哪里出问题，只是默默返回一个空串，你的程序拿着空串兴冲冲地去连接数据库，结果连接失败、日志满屏「dial tcp: missing address」，排查半天才反应过来是环境变量根本没设置。建议师弟在所有关键配置的读取点，都用 `LookupEnv` 并在布尔值为 `false` 时立即 panic 或打印醒目错误，防止这种藏得很深的故障。

### os.Environ：把全家福都端出来

小师弟，有时候你不只想查某一个环境变量，而是想把当前进程的「家底」一次性都翻出来看看。这时候就轮到 `os.Environ()` 登场了。

`os.Environ()` 的返回值是一个 `[]string`，切片里的每个元素都长成 `KEY=VALUE` 这副模样，比如 `PATH=/usr/local/bin:/usr/bin` 或者 `HOME=/root`。它拍下的是当前进程所有环境变量的一张**快照**——调用之后再往环境里加新变量，这份切片是不会自动更新的，请牢记这一点。

来看一段实际代码：

```go
package main

import (
    "fmt"
    "os"
)

func main() {
    envs := os.Environ()
    for _, entry := range envs {
        fmt.Println(entry)
    }
}
```

逻辑很简单：调用 `os.Environ()` 拿到切片，再用 `for range` 逐条打印。每一行输出就是一个 `KEY=VALUE` 字符串，如果你想进一步拆分，可以用 `strings.SplitN(entry, "=", 2)` 把键和值分开，注意用 `2` 作为分割上限，因为值本身也可能包含等号。

这里有两个坑要提醒你。**第一个坑是大小写问题。** 在 Unix / Linux 系统下，环境变量名按约定全部大写，`Path` 和 `PATH` 是两个不同的变量；但在 Windows 里，环境变量名不区分大小写，系统会帮你做统一处理。如果你写的程序要跨平台运行，在比较变量名时一定要注意这个差异，别踩坑。**第二个坑是遍历顺序。** `os.Environ()` 返回的切片顺序并不保证稳定，每次运行结果的排列可能都不一样，所以不要依赖顺序来做任何业务逻辑判断。

总体来说，`os.Environ()` 是调试和运维场景下非常顺手的工具，把全家福端出来一眼扫清，省心又省力。

### strings.SplitN：按等号切两刀

师弟，你有没有想过，为什么示例代码里用的是 `strings.SplitN(e, "=", 2)`，而不是更常见的 `strings.Split(e, "=")`？这里有一个很容易踩坑的细节，师兄今天专门讲清楚。

先说 `strings.Split` 的行为：它会把字符串按照分隔符**全部切开**，有几个等号就切几刀，返回的切片长度不可预测。这在处理普通字符串时没什么问题，但环境变量不一样——环境变量的**值本身就可能包含等号**。

举个真实的例子。假设你的系统里有这样一条环境变量：

```
DATABASE_URL=postgres://u=p@host/db
```

这里的值 `postgres://u=p@host/db` 中间藏了一个等号，用来表示用户名和密码的连接形式。如果你用 `strings.Split(e, "=")` 来处理这个字符串，你会得到三段：

```go
// 结果：["DATABASE_URL", "postgres://u", "p@host/db"]
```

完全乱掉了，key 倒是对的，但 value 被切碎了，后续代码拿到的数据根本没法用。

而 `strings.SplitN(e, "=", 2)` 的第三个参数 `n=2`，意思是**最多返回 2 段**，也就是只在**第一个等号**处切一刀，剩下的内容不管有没有等号，统统归入第二段：

```go
parts := strings.SplitN("DATABASE_URL=postgres://u=p@host/db", "=", 2)
// 结果：["DATABASE_URL", "postgres://u=p@host/db"]
```

这样 `parts[0]` 稳稳地拿到 key，`parts[1]` 完整地保留了 value，一个字符都不会丢。

记住这个原则：**凡是解析 key=value 格式的字符串，优先用 `SplitN` 并限制段数为 2**，这是处理此类结构最安全、最地道的写法，师兄在生产代码里吃过亏才总结出来的经验，师弟你直接收下就好。

### 生产实战：环境变量是配置的好朋友

师弟，你现在写的那些数据库密码、API Key、JWT 密钥，千万别直接硬编码进代码里。一旦推送到 GitHub，密钥就等于公开了，后果相当严重。师兄的建议是：把这些敏感配置全部放进**环境变量**，这是业界最普遍、也最实用的做法。

环境变量有两个核心优势。第一，**跨环境切换极其方便**。本地开发、测试服务器、生产服务器往往连接不同的数据库，只需在各自的机器上设置不同的环境变量值，代码本身一行都不用改。第二，**密钥不会进入代码仓库**。你的 `.go` 文件里只会出现 `os.Getenv("DB_PASSWORD")` 这样的读取语句，真正的密码永远留在运行环境里，Git 历史记录里干干净净。

不过师兄也要提醒你一个容易踩的坑：**日志里绝对不要打印全量环境变量**。有些同学调试时图省事，直接把 `os.Environ()` 的输出全部打进日志，这一下子就把所有密钥都暴露在日志文件里了，风险极大。日志只记录你需要的业务信息，密钥相关的内容一律不打。

另外，本地开发时手动设置环境变量有点繁琐，社区有个叫 **godotenv** 的库，可以配合项目根目录下的 `.env` 文件使用，启动时自动加载，记得把 `.env` 加进 `.gitignore`。等你对项目配置管理更熟悉之后，可以深入了解一下，会让你的开发体验好很多。

`FOO` 已在代码里用 `os.Setenv` 设为 `1`，所以能读到；`BAR` 没有设置，`os.Getenv` 返回空串，输出为空。最后遍历打印的环境变量名列表**因机器、用户、shell 配置不同而不同**，你的终端里看到的条数可能比示例多很多，不必担心。

**方式二：在命令前临时注入环境变量**

```shell
BAR=2 go run environment-variables.go
```

```text
FOO: 1
BAR: 2
```

在命令前加 `KEY=VALUE` 是 bash/zsh 的语法，只对这一次子进程生效，不会污染当前 shell 的全局环境，是临时调试的好习惯。

**方式三：用 export 先设再跑**

```shell
export BAR=hello
go run environment-variables.go
```

`export` 会把变量写入当前 shell，后续所有子进程都能继承，适合需要多次复用同一配置的场景。测试完记得 `unset BAR` 清理现场。

## 💡 师兄的碎碎念

**师兄碎碎念**

读取环境变量时，`os.Getenv` 在变量未设置和变量被设为空字符串这两种情况下都返回空字符串，无法区分。如果你的业务逻辑需要判断「这个变量到底有没有被配置」，请改用 `os.LookupEnv`，它会额外返回一个布尔值，`false` 就表示该变量根本不存在于环境中，语义更严谨，也能避免把「未配置」和「故意置空」混为一谈。

跨平台时要注意设置环境变量的语法差异：Unix/macOS 用 `export KEY=value`，而 Windows CMD 用 `set KEY=value`，PowerShell 则是 `$env:KEY = 'value'`。在 CI 脚本里写死某一种语法会让另一个平台的同学踩坑，建议在 Makefile 或脚本里做好平台判断，或者统一用跨平台工具链。

安全上有一条铁律：**日志里绝对不要打印全量环境变量**。`os.Environ()` 返回的列表往往混有数据库密码、API 密钥、Token 等敏感信息，一旦写入日志系统就可能大面积泄露，排查问题时只打印你明确需要的那几个键即可。

本地开发时，推荐使用社区知名库 `godotenv`，它可以把 `.env` 文件中的键值对自动注入到进程环境，省去每次手动 `export` 的麻烦，记得把 `.env` 加入 `.gitignore`，切勿提交到代码仓库。

最后，容器化和云原生场景里有一个约定俗成的规范：环境变量命名应使用**全大写字母加下划线**分隔，例如 `DATABASE_URL`、`MAX_RETRY_COUNT`，这样既符合 Twelve-Factor App 的最佳实践，也方便运维在 Kubernetes 的 `env` 字段中统一管理配置。
